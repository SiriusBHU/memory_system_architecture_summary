## 框架总览：端侧内存管理全景与七个洞察

下图把**端侧内存管理系统的全貌**与本调研涉及的 **SoC 组件**画在一张图上（应用/Agent 层 → OS 内核内存管理 → SoC 计算 IP / 内存子系统 / 存储，安全横切其间，云侧大脑在顶层做端云协同），并把七个洞察按编号标注到各自所处的位置。

![端侧内存管理系统全景与七个洞察](assets/n1-framework.svg)

下面分块说明：每一块**具体面临什么问题**，以及对应洞察**怎么解**（编号与图中一致）。

**① KV Cache 分级管理**（应用层 ↔ 存储层）
- 问题：长上下文 KV Cache 随序列线性膨胀（32K 上下文单请求约 4.3 GB），超出端侧 DRAM；多 Agent 切换需冷重算，TTFT 高达 15–172 s。
- 洞察解法：KV Cache 三级分级（DRAM 热 → NVMe 温 → 3D NAND 冷）+ Q4 量化（压到约 28% 体积）+ 持久化跳过 prefill（172 s → 1.8 s，约 94×）+ KVNAND 闪存内注意力计算，把「一个上下文=一份 DRAM 拷贝」打破。

**② 端侧可编程 / 自适应内存**（OS 内核内存管理）
- 问题：内核策略固定、一刀切——全有或全无的 THP 在长跑下碎片化（运行 2 h 后大页分配成功率约 50% → <10%）；固定 LMKD 阈值要么杀晚（卡顿）要么杀早（冷启动慢）。
- 洞察解法：把策略做成可编程 / 自适应——多尺寸 mTHP、DAMON 访问监控驱动主动回收（省 32% 内存 @ 约 1.9% 开销）、SWAM 自适应换页+杀进程（OOM kill 降约 6.5×）、eBPF 钩子让内存策略可在端侧按负载定制。

**③ 内存带宽：供给 + 利用**（SoC 计算 IP ↔ 内存子系统）
- 问题：端侧解码受内存带宽限——LPDDR5 约 51 GB/s 把 7B 卡在约 14 tok/s；且单一加速器只能用满约 60% 的 DRAM 总线。
- 洞察解法：供给侧把管子加宽（LPDDR5X +67% 带宽、2× 容量、−25% 功耗），利用侧把宽管榨满（CPU/GPU/NPU 异构协同，预填充→NPU、解码→GPU，利用率约 60% → 88%，reactive 延迟 −4.6×）。

**④ 端侧分级内存**（OS 内核内存管理：压缩内存 ↔ swap）
- 问题：DRAM 有界且共享，通用 LRU 回收要么杀进程、要么卡顿；冷页白占 DRAM，GB 级 App 冷启动频发（86.6% 越过 1 s 悬崖）。
- 洞察解法：DRAM ↔ 压缩内存（zram/zswap，约 2:1）↔ Flash 三级分级 + MGLRU 冷热分代（ChromeOS 上 kswapd CPU −40%、低内存杀进程 −85%）+ AppFlow 预测式预取/回收（冷启动 2 s → 690 ms）。

**⑤ ARM 内存安全 / 机密计算**（横切全栈数据面）
- 问题：端侧内存型漏洞（UAF / 越界）频发，纯软件防护开销大；端侧 ML 模型、KV 与记忆数据、生物识别等敏感工作负载需要硬件级隔离。
- 洞察解法：ARM MTE 硬件内存标签（低开销实时抓内存错误）+ CCA/RME 与 pKVM 机密计算，为端侧机密工作负载（ML 推理 / DRM / 生物识别）提供硬件隔离，贯穿应用→内核→内存每一层。

**⑥ Agent 外置分层记忆系统**（应用 / Agent 层）
- 问题：上下文窗口有限，每轮重放全历史使 token >10×、p95 延迟约 11×；截断历史又导致 Agent 遗忘。
- 洞察解法：把记忆外置为分层记忆 OS（STM→MTM→LPM），活跃工作集封顶（MemoryOS 仅 7 页），用按需检索代替全量重放（token −70~90%，LoCoMo 精度 +26~49%）。

**⑦ 端云协同 · 千人千面冷热识别**（云侧大脑 ↔ 端侧全栈）
- 问题：设备本地通用冷热策略一刀切，对每个用户/场景预测错（86.6% 的 GB 冷启动越过 1 s 悬崖）；而端侧学习器数据/算力/标签都不足，独立学不出强的个性化预测器。
- 洞察解法：端侧「小脑」（个性化场景冷热预测器）+ 云侧「大脑」（聚合全舰队场景、训练并下发千人千面策略）+ 智能请求门控（仅约 5% 调云、原始行为留本地），把冷热识别做到「千人千面」。
