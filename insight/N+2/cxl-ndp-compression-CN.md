# CXL 近数据处理与内存压缩

> **原始方案：** 标准 CXL.mem —— 主机管理的内存扩展，设备端不做任何处理，数据以原始宽度在 CXL 链路上全量传输。
> **演进方案：** CXL + NDP + 压缩 —— 设备端压缩/解压、位平面重组、CXL Type-3 设备内透明带宽放大。

---

## 2. 问题背景

CXL（Compute Express Link）通过基于 PCIe 的一致性互连将 DDR 内存池接入服务器，实现字节可寻址的内存扩展。CXL Type-3 设备无需专用驱动即可向主机暴露大容量内存。CXL 3.0（2023，多级交换）和 CXL 4.0（2025 年 11 月，128 GT/s，端口捆绑）的相继发布加速了产业落地。

核心矛盾在于：**容量可以扩展，但带宽跟不上。** DDR5-6400 单通道仅提供 51.2 GB/s；PCIe 5.0 x16 CXL 链路峰值约 64 GB/s。LLM 推理——尤其是带有大规模 KV 缓存的解码阶段——属于典型的内存带宽受限场景。将 KV 缓存卸载到 CXL 可以缓解 GPU HBM 压力，但瓶颈随之转移到 CXL 链路本身。实测 CXL.mem 访问延迟约 640 ns，加之有限的链路带宽，意味着朴素的 CXL 内存扩展非但无法加速推理，反而可能导致停顿。因此问题变为：设备自身能否在数据到达链路之前减少传输量？

---

## 3. 具体问题与实证

### 3.1 CXL 链路带宽成为新瓶颈

标准 CXL.mem 逐字节原样搬运所有请求数据。对于一个 70B 参数 BF16 模型，即使有效精度仅需 5 位，单次 16 KB 权重读取仍在链路上传输完整的 16 KB。上下文长度超过 64k token 后，仅 KV 缓存流量就能打满链路。

**实证 (S1)：** 在 H100 + 256 GB CXL 上部署 LLaMA 3.1 70B 时，被动 CXL 方案的吞吐在约 65k-token 上下文处触顶；此后解码阶段完全受限于链路带宽。

### 3.2 字节对齐格式浪费 DRAM 带宽

按标准字节序存储的浮点值可压缩性很低。对原始 BF16 权重张量直接运行 LZ4/ZSTD 仅能压缩 0–23%；原始 KV 缓存块的压缩率更低，仅 0.9–6.5%。

**实证 (S1)：** 对未经重组的 LLaMA 3.1 8B BF16 权重直接施加传统字节级压缩，收益微乎其微——原因在于每个值的指数位与尾数位交错存储，破坏了值间的规律性。

### 3.3 块级压缩中的提升/降级开销

将数据区分为热数据（不压缩）和冷数据（压缩）的块级内存压缩方案，在降级（重新压缩刚被提升的块）时存在显著性能损耗。

**实证 (S2)：** 最新的基于提升的方案在降级周期中产生高达 40% 的元数据访问开销，导致内存密集型工作负载的有效加速比低于 1.2 倍。

### 3.4 解聚合推理中的 KV 缓存召回代价

非驱逐型 KV 缓存框架将整个缓存卸载到 CXL 内存，但在执行注意力计算时须将 token 页面召回 GPU 显存。在百万 token 上下文下，召回流量成为主导。

**实证 (S3)：** 在百万 token 场景下，无近存处理的基准 CXL KV 卸载方案中，召回带宽占用了可用 CXL 链路容量的 85% 以上，几乎没有为权重预取留出余量。

## 4. 架构：原始方案 vs. 演进方案

### 4.1 原始方案：标准 CXL.mem（被动扩展）

```
 ┌──────────────────────────────────────────────────┐
 │                     主机 (HOST)                    │
 │  ┌──────────┐    ┌──────────┐    ┌───────────┐   │
 │  │   CPU    │    │   GPU    │    │ 应用/LLM   │   │
 │  │ (内存    │◄──►│  (HBM)   │◄──►│  运行时    │   │
 │  │ 控制器)  │    └──────────┘    └───────────┘   │
 │  └────┬─────┘                                     │
 │       │  CXL.mem（原始、未压缩）                    │
 │───────┼───────────────────────────────────────────│
 │       ▼                                           │
 │  ┌────────────────────────────────────────┐       │
 │  │         CXL Type-3 设备                │       │
 │  │  ┌─────────────┐   ┌───────────────┐  │       │
 │  │  │ CXL.mem     │──►│  DDR 内存     │  │       │
 │  │  │ 控制器      │   │  控制器       │  │       │
 │  │  │ (直通)      │◄──│  (标准)       │  │       │
 │  │  └─────────────┘   └──────┬────────┘  │       │
 │  │                           │            │       │
 │  │                   ┌───────▼────────┐   │       │
 │  │                   │  DRAM 通道     │   │       │
 │  │                   │  (DDR5, 原始)  │   │       │
 │  │                   └────────────────┘   │       │
 │  └────────────────────────────────────────┘       │
 └──────────────────────────────────────────────────┘

 数据流：主机请求 16 KB → 设备从 DRAM 读取 16 KB
         → 通过 CXL 链路发送 16 KB。
 无任何变换。链路带宽 = 有效带宽。
```

### 4.2 演进方案：CXL-NDP + 压缩 + 位平面布局

```
 ┌───────────────────────────────────────────────────────┐
 │                       主机 (HOST)                      │
 │  ┌──────────┐   ┌──────────┐   ┌─────────────────┐   │
 │  │   CPU    │   │   GPU    │   │  应用/LLM       │   │
 │  │ (可见   │◄─►│  (HBM)   │◄─►│  运行时         │   │
 │  │ 多个逻辑│   └──────────┘   │ (精度感知区域)   │   │
 │  │ 区域)   │                  └─────────────────┘   │
 │  └────┬─────┘                                        │
 │       │  CXL.mem（压缩后，流量大幅减少）               │
 │───────┼──────────────────────────────────────────────│
 │       ▼                                              │
 │  ┌──────────────────────────────────────────────┐    │
 │  │          CXL Type-3 设备 + NDP 控制器        │    │
 │  │                                              │    │
 │  │  ┌──────────────┐                            │    │
 │  │  │ 请求前端      │  解析 CXL.mem 读写请求，  │    │
 │  │  │ (64-entry    │  识别精度区域，发出         │    │
 │  │  │  MSHR)       │  平面级请求                 │    │
 │  │  └──────┬───────┘                            │    │
 │  │         ▼                                    │    │
 │  │  ┌──────────────┐                            │    │
 │  │  │ 平面索引      │  2 MB SRAM 查找表：       │    │
 │  │  │ 与元数据      │  块 → 平面 ID、DRAM 行、  │    │
 │  │  │              │  编解码标签                 │    │
 │  │  └──────┬───────┘                            │    │
 │  │         ▼                                    │    │
 │  │  ┌──────────────┐                            │    │
 │  │  │ 编解码器      │  32 并行通道，             │    │
 │  │  │ 阵列          │  聚合吞吐 2 TB/s，        │    │
 │  │  │ (LZ4/ZSTD)  │  压缩 + 解压               │    │
 │  │  └──────┬───────┘                            │    │
 │  │         ▼                                    │    │
 │  │  ┌──────────────┐  ┌──────────────────────┐ │    │
 │  │  │ DRAM         │  │  位平面布局           │ │    │
 │  │  │ 调度器       │──│  P^sgn │ P^exp_0..E  │ │    │
 │  │  │ (FR-FCFS,   │  │  P^man_0..M           │ │    │
 │  │  │  行感知)     │  │  (每个值的各位按平面   │ │    │
 │  │  └──────────────┘  │   分组存储)            │ │    │
 │  │                    └──────────────────────┘ │    │
 │  └──────────────────────────────────────────────┘    │
 └───────────────────────────────────────────────────────┘

 数据流：主机请求 16 KB（5 位精度）→
   设备仅读取符号位 + 前 5 个平面（约为 DRAM 读取量的 5/16），
   解压后还原为 16 KB 通过 CXL 链路返回。
 有效带宽放大倍数：相比原始链路可达 2.5–3.2 倍。
```

## 5. 解决了什么 / 没解决什么

### 解决的问题

| 方面 | 机制 |
|------|------|
| **CXL 链路带宽** | 压缩后的数据经链路传输；有效带宽按压缩比放大（1.3–1.9 倍）。 |
| **DRAM 访问能耗** | 位平面选择性读取减少行激活次数；最高可降低 40.3% 能耗 (S1)。 |
| **KV 缓存占用** | 设备端压缩将 KV 存储缩减 44.8–46.9%，且无精度损失 (S1)。 |
| **上下文长度上限** | 释放的容量将最大上下文扩展 87%（65k → ~122k 可用 token, S1）。 |
| **模型加载延迟** | 压缩后的权重加载更快；延迟最高降低 42.1% (S1)。 |
| **冷数据容量** | IBEX 风格的提升式压缩将冷块有效容量扩展约 1.59 倍 (S2)。 |
| **召回流量 (PNM)** | 近存处理完成 token 页面选择，消除 GPU 召回；吞吐提升最高 21.9 倍 (S3)。 |

### 未解决的问题

| 方面 | 原因 |
|------|------|
| **CXL 访问延迟** | 压缩/解压增加约 25–100 ns 编解码延迟；CXL 基础往返延迟（~640 ns）不变。 |
| **已量化模型** | 当基础权重已为 INT4/FP8 时，位平面压缩额外收益仅 1.9–2.1% (S1)。 |
| **写密集型工作负载** | KV 缓存写入需要缓冲（256-token 窗口）+ 压缩后才能提交 DRAM；存在写放大风险。 |
| **主机软件适配** | 精度分区地址区域需要页表配置；虽透明但非零成本集成。 |
| **设备间交换网络** | 多设备 CXL 交换网络的延迟和争用与设备内部 NDP 正交。 |
| **GPU 算力瓶颈** | NDP 优化的是带宽而非算力；在注意力计算受限的场景下收益递减。 |

## 7. 一词判定

**值得推进（Compelling）。** —— 设备端 NDP 与压缩将 CXL 链路从纯容量扩展手段转变为带宽放大型内存层级，在 LLM 推理最关键的指标上取得 40–87% 的提升，且硅片成本可控。

## 8. 开放问题

1. **编解码器标准化。** CXL-NDP 和 IBEX 各自定义了专有的编解码流水线。CXL 5.0 或 OCP 子规范是否会标准化内联压缩接口，使不同厂商的设备具备互操作性？

2. **写路径放大。** KV 缓存压缩需要缓冲（CXL-NDP 中为 256-token 窗口）后才能提交 DRAM。在突发式解码模式下，需要多大的 SRAM 缓冲才能避免停顿？能耗代价如何？

3. **多设备一致性。** CXL 4.0 支持多机架内存池化。当各设备的 NDP 控制器独立压缩数据时，交换网络层面的一致性和迁移协议如何处理异构压缩状态？

4. **量化后的收益递减。** 对已量化（INT4/FP8）模型施加位平面压缩仅额外节省不足 3%。随着模型量化成为常态，设备端压缩是否仍具成本合理性？

5. **安全与侧信道。** 压缩比与数据相关。在多租户部署中，CXL 链路流量或延迟的可观测变化是否会泄露模型权重或用户提示词的信息？

6. **与结构化稀疏的协同。** 稀疏 MoE 和剪枝模型具有不规则访问模式。位平面布局和块级压缩对高稀疏性张量的适应性如何？
