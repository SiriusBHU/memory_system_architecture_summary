# A08 · 压力感知与低内存终止（PSI / LMKD / Jetsam）

> **一句话定位**：当回收（[A4](A04-回收总论.md)）也救不回内存时，系统靠什么"感知到撑不住了"，又按什么顺序"主动杀掉一些进程"来保命——这是内存子系统的最后一道安全阀。
>
> 📍 **对应总览**：[00 总览](00-内存系统总览.md) 的「2B 物理与回收侧」最末环节——压力感知与低内存终止（PRESS 格）。
> 🧭 **阅读前置**：先读 [A4 回收总论](A04-回收总论.md)（终止是回收失败后的兜底）；与 [A7 cgroup/memcg](A07-cgroup-memcg.md) 对照（memcg 是**局部限额**，本篇是**全局救火**）；[A9 设备内存全景](A09-设备内存全景.md) 解释 pinned 页如何加剧压力。
> 🌡️ **演进分级**：**演进厚 ⚡**——感知侧从"水线/vmpressure"演进到 **PSI**，终止侧从**内核态 LMK** 迁到**用户态 LMKD**，是近十年移动内存最显著的一条架构迁移线。**重点在 §4 历史与 §6 趋势**。

## 1. 定位：它在地图上的哪一格

回收（A4）的工作是"腾出页"；但回收有上限——当**回收速度追不上分配速度**，或干脆**没有可回收的页**（全是匿名脏页又无 swap、全是 pinned 设备内存），系统就进入真正的低内存危机。这一格回答两件事：

1. **感知**：怎么*量化*"内存有多紧张"，且要在系统彻底卡死*之前*就知道；
2. **终止**：危机确认后，按什么*优先级*杀进程释放内存，杀谁、谁来杀。

注意术语严格区分，三套机制分属不同平台与层次，**切勿混用**：

| 机制 | 平台 / 层次 | 角色 |
|---|---|---|
| **PSI** + watermark 水线 | Linux 内核（感知） | 量化压力、向上报信 |
| **LMK → LMKD** | Android（终止） | LMK 是已废弃的*内核驱动*；LMKD 是当前的*用户态守护进程* |
| **Jetsam / memorystatus** | iOS / Darwin（终止） | XNU 内核线程，按优先级带杀进程 |
| **OOM Killer** | Linux 内核（终止·兜底） | 最后一道防线，谁都没拦住时由内核动手 |

## 2. 它解决什么问题

移动设备没有（或极少用）磁盘 swap，匿名页无处可去，一旦内存见底，朴素做法是放任内核 **OOM Killer** 在"分配彻底失败"那一刻才动手。但那太晚了：在触发 OOM 之前，系统往往已**长时间颠簸（thrashing）**——CPU 大部分时间耗在反复换入换出 page cache 上，界面早已卡死，用户体验崩塌却一个进程都没被杀。

所以现代系统需要一个**更早、更平滑的压力信号**，在"还能动"时就开始有选择地终止后台进程。这正是 PSI 与用户态 LMKD/Jetsam 要解决的核心矛盾：**把"什么时候该杀、该杀谁"的决策从内核的硬阈值，挪到能感知真实停顿、可按业务优先级裁决的位置。**

## 3. 机制本体：当前是怎么做的

### 3.1 感知（其一）：watermark 水线——快但粗

内核伙伴系统为每个 zone 维护三条水线 `min` / `low` / `high`（详见 A3）。空闲页跌破 `low` 唤醒 **kswapd** 后台回收；跌破 `min` 触发**直接回收**（分配者自己同步回收，会阻塞）；`min` 也守不住，就走向 OOM。水线是**对"空闲页绝对量"的瞬时判断**——简单、低开销，但**它只看"还剩多少页"，看不到"为这些页付出了多少停顿代价"**。一台设备可能空闲页不低，却因反复 refault 而卡顿；水线对此一无所知。

### 3.2 感知（其二）：PSI——量化"为内存付出的停顿"

**PSI（Pressure Stall Information，压力停顿信息）** 由 Johannes Weiner 开发，**在 Linux 4.20 合入主线**（`CONFIG_PSI`）。它换了一个根本视角：不看"剩多少页"，而是**统计"任务因等待资源而被卡住的墙钟时间占比"**。([PSI 文档](https://docs.kernel.org/accounting/psi.html)，[Tracking pressure-stall information, LWN](https://lwn.net/Articles/759781/))

PSI 覆盖三类资源，导出在 `/proc/pressure/{cpu,memory,io}`，每类给出两种粒度：

- **`some`**：至少有一个任务因该资源而停顿的时间占比——反映"有没有人受影响"；
- **`full`**：所有非空闲任务**同时**都因该资源停顿的时间占比——此时 **CPU 周期纯属浪费**，是系统**颠簸（thrashing）**的硬指标。（注：CPU 在系统级无 `full`，该行自 5.13 起才有定义。）

每个指标给出 10s / 60s / 300s 三个滑动窗口的平均，兼顾短期突发与中长期趋势。更关键的是 **PSI 监视器（trigger）API**：特权用户态进程可向 `/proc/pressure/memory` 写入"窗口内停顿阈值"，再用 `poll()` 订阅——一旦内核测到停顿超阈，立即唤醒该进程。窗口可取 **500ms～10s**。这把"压力感知"变成了**事件驱动**，正是 LMKD 的信号来源。([PSI 文档](https://docs.kernel.org/accounting/psi.html))

### 3.3 终止：Android 的 LMKD

**LMKD（Low Memory Killer Daemon，低内存终止守护进程）** 是 Android 的用户态进程，订阅 PSI 监视器事件。被唤醒后，它读取系统内存状态，并按进程的 **`oom_score_adj`** 决定杀谁——这个值由 `ActivityManagerService` 按进程的可见性/重要性动态设定：前台 App 接近 0（甚至负值，几乎不被杀），各级缓存的后台 App 取较高的正值。**`oom_score_adj` 越高 = 越不重要 = 越先被杀**。([Low memory killer daemon, AOSP](https://source.android.com/docs/core/perf/lmkd))

LMKD 把压力分三档，每档对应一个最小 `oom_score_adj` 门槛，只杀"分数 ≥ 该门槛"的进程：

| 压力档 | 行为 | 默认门槛（`oom_score_adj`） |
|---|---|---|
| **low** | 通常禁用 | ≥ 1001（即不杀） |
| **medium** | 杀缓存的、非必要的后台进程 | ≥ 800 |
| **critical** | 必要时杀任意进程 | ≥ 0 |

被杀进程优先选 `oom_score_adj` 最高者；同档内还会参考 RSS 等因素以最大化单次回收量。LMKD 仍保留一个**兼容旧行为的 minfree 模式**（`ro.lmk.use_minfree_levels`），按"空闲内存 + 文件缓存"水线杀进程，对应早年内核 LMK 的判据；但 Android 11+ 主推 PSI 路线。([Low memory killer daemon, AOSP](https://source.android.com/docs/core/perf/lmkd))

### 3.4 终止：iOS 的 Jetsam / memorystatus

iOS 没有这套用户态守护进程，终止逻辑**在内核内**。XNU 的 **memorystatus** 子系统跑一个内核线程，因 iOS **基本不用磁盘 swap**（匿名页只能靠压缩内存兜，见 A6），内存见底时只能"丢"——这套"丢进程"的机制俗称 **Jetsam**。它维护一张全进程的 **jetsam 优先级带（priority band）** 列表，当可回收页跌破阈值时，**按优先级升序**（最不重要的带最先）杀进程；同时每个进程有**内存上限（memory limit）**，超过**软上限**只告警、超过**硬上限**则直接被终止。([memorystatus_notify.md, apple-oss-distributions/xnu](https://github.com/apple-oss-distributions/xnu/blob/main/doc/vm/memorystatus_notify.md))

memorystatus 另有一套**压力等级**通知（对照 Android 三档）：**Normal / Warn(Urgent) / Critical**，阈值按"未压缩可用内存占总量比例"划定（典型 Warn ≈ 0.5×、Critical ≈ 0.29–0.4× 可用内存，随平台而异）；App 收到 Warn/Critical 后应主动释放缓存。Jetsam 维持的"可回收页池"目标约为总内存的 **5%/10%/15%** 量级。([memorystatus_notify.md, XNU](https://github.com/apple-oss-distributions/xnu/blob/main/doc/vm/memorystatus_notify.md))

### 3.5 兜底：内核 OOM Killer

无论哪个平台，若上层都没拦住、分配在 `__alloc_pages_slowpath()` 中直接回收也失败，内核会调用 `out_of_memory()`。`oom_badness()`（`mm/oom_kill.c`）给每个候选进程算一个 0～1000 的"badness"分：基础项是 `rss + swap + 页表页`，再叠加 `oom_score_adj` 的比例修正（同一个旋钮在这里也生效，`-1000` 表示永不杀），**选分最高者杀掉**。OOM Killer 是**最后一道防线**——在移动端，理想情况下 LMKD/Jetsam 早已介入，它几乎不该被触发。([proc_pid_oom_score_adj(5), man7](https://man7.org/linux/man-pages/man5/proc_pid_oom_score_adj.5.html))

### 3.6 进程被杀后：内存怎么还回去

无论谁动的手，被杀进程退出时，内核 `exit_mm()` 回路会**逐项归还其占用**：

- **匿名页**（堆、栈、`.bss`）→ 解除映射后释放回伙伴系统；若在 zram/zswap 里则释放压缩槽（A6）；
- **页表**（PGD~PTE 各级）→ 拆除并回收（A1）；
- **文件页 / page cache**：该进程**私有映射**的部分随之解映射，但**共享的 page cache 不会立即丢**——它仍可能被别的进程引用或留作缓存，由回收路径按 LRU 决定去留（A4/A11）；
- **设备内存**：进程持有的 dma-buf / GPU buffer 引用计数减一，**只有归零才真正还页**——这正是 pinned 设备内存的麻烦所在（见 §7、A9）。

> 一个常被忽视的点：**杀一个进程释放的内存，远不止它的 RSS**。它的页表、它独占的缓存、它 pin 住的设备 buffer 一并松绑，这也是 LMKD/Jetsam 倾向于"杀一个大的"而非"杀一堆小的"的原因。

## 4. 历史：为什么演变成今天这样 ← 本篇重点

终止机制的演进，是一条**从内核硬阈值 → 用户态智能决策**的迁移线。

**第一代·内核态 LMK（lowmemorykiller 驱动）**。早期 Android 在内核里塞了一个 `lowmemorykiller` 驱动：配一组"空闲内存 + 文件缓存"水线（minfree），跌破某档就在内核里直接挑 `oom_score_adj` 最高的进程杀掉。它的致命缺陷有二：(1) **只看空闲量，不看压力**——低内存设备必须把水线调得很激进，否则面对大量 file-backed active page cache 时会**只颠簸、不杀进程**，性能崩了却迟迟不动手；(2) 内核态策略**死板**，难以表达"按 App 业务重要性裁决"这类用户态才懂的语义。([Low memory killer daemon, AOSP](https://source.android.com/docs/core/perf/lmkd))

**迁移的前提·把策略搬到用户态**。**内核 LMK 驱动在 Linux 4.12 被移出主线**，监控与终止的职责整体交给用户态 **LMKD**。但 LMKD 初期只能用 **`vmpressure`** 信号（基于回收扫描效率的粗估），**误报极多**——回收扫得多不代表用户真在受苦。([Low memory killer daemon, AOSP](https://source.android.com/docs/core/perf/lmkd))

**第二代·LMKD + PSI（Android 10 起）**。PSI 在内核 4.20 合入（并被回移植到 Android 常用的 4.9 / 4.14 内核），给了 LMKD 真正想要的信号：**直接量化"任务为内存等了多久"**。**Android 10 引入 LMKD 的 PSI 模式**，用 PSI 监视器替代 `vmpressure`，按 `some`/`full` 停顿阈值分档触发，误报大降、介入时机更准。这一步完成了"**感知**（PSI，内核给出客观停顿）+ **决策**（LMKD，用户态按 `oom_score_adj` 裁决）"的职责分离——正是今天的形态。([Low memory killer daemon, AOSP](https://source.android.com/docs/core/perf/lmkd)，[PSI 文档](https://docs.kernel.org/accounting/psi.html))

> PSI 本身的动机也印证了这条线：LWN 在 PSI 合入时即点明，它"已在 Facebook 内部用于优化资源分配"，且"Android 圈很关注——开发者想在系统性能彻底崩塌*之前*更早地发现 OOM 苗头"。([Tracking pressure-stall information, LWN](https://lwn.net/Articles/759781/))

iOS 侧没有经历这种"内核→用户态"的迁移：Jetsam 自始至终在 XNU 内核内，靠 memorystatus 的优先级带模型运作。两条路线殊途同归——**都用优先级把"该杀谁"显式化**，差别在 Android 把决策放到了用户态、iOS 留在内核态。

## 5. 平台差异（横切）

| 维度 | Android | iOS / Darwin | HarmonyOS |
|---|---|---|---|
| 感知信号 | **PSI**（10+）+ watermark；旧版 vmpressure | memorystatus 压力等级（Normal/Warn/Critical） | 待核实（Linux 基座设备可用 PSI） |
| 终止执行者 | **用户态 LMKD** | **内核 Jetsam / memorystatus** | 待核实 |
| 优先级旋钮 | `oom_score_adj`（AMS 动态设定） | jetsam priority band + per-proc memory limit | 待核实 |
| swap | 多用 zram（A6），少用磁盘 swap | 基本无磁盘 swap，仅压缩内存 | 待核实 |
| 内核兜底 | Linux OOM Killer | XNU OOM（极少触达） | 待核实 |

## 6. 趋势与未解问题 ← 本篇重点

- **PSI 成为通用压力总线**：PSI 已不止服务 LMKD——容器编排（Kubernetes 读 PSI 做调度与驱逐）、`systemd-oomd`（桌面 Linux 用 PSI 触发用户态 OOM）、以及 per-cgroup PSI（每个 memcg 单独读压力，把"全局救火"细化到"每组救火"，与 [A7](A07-cgroup-memcg.md) 的限额体系合流）都在用它。压力感知正从"一个全局数"走向"可按作用域订阅的事件流"。
- **更智能的杀进程时机**：纯阈值仍偏被动；趋势是结合 refault/workingset（A5）与停顿趋势做**预测性终止**，在颠簸*形成前*就出手。
- **设备内存让"杀进程"越来越不灵**：GPU/相机/AI 的 dma-buf、CMA、secure heap 多为 **pinned 不可回收**（A9）。当压力主要来自这类页时，**杀普通后台进程根本释放不出它们**——除非杀掉持有引用的那个进程。如何把"设备内存压力"也并入 PSI/优先级模型，是开放问题。
- **可计量性与归因**：被杀进程释放的内存难以精确预估（共享缓存、跨进程 dma-buf 引用），导致 LMKD/Jetsam 有时"杀了还不够、又得再杀一轮"。更准的 per-process 可回收量计量（A13）仍在演进。

## 8. 实测 / 观测点

- `cat /proc/pressure/memory`：看 `some`/`full` 的 10s/60s/300s 停顿占比——`full avg10` 持续偏高 = 正在颠簸；
- `cat /proc/<pid>/oom_score_adj` 与 `oom_score`：看某进程的"可杀程度"；Android 上对照 `dumpsys activity oom` 看各 App 的 adj 分档；
- Android：`logcat` 里 `lowmemorykiller` / `lmkd` 标签可见每次终止的进程、原因、当时压力档；`dumpsys meminfo` 看全局水位；
- iOS：设备「设置 → 隐私 → 分析与改进」里的 **JetsamEvent** 报告，记录每次 Jetsam 的优先级带、各进程页数与触发原因；
- `dmesg` 里的 `Out of memory: Killed process …` 是内核 OOM Killer 真正动手的证据——移动端正常情况下罕见，频繁出现说明上层终止没跟上。
