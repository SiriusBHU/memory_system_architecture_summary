# A01 · 地址空间与虚实转换

> **一句话定位**：虚拟内存系统的"地基"——一个虚拟地址如何被*描述*（VMA）、被*记录*（页表）、被硬件*翻译*（MMU/TLB），以及用户态如何进出这套机制。
>
> 📍 **对应总览**：[00 总览](00-内存系统总览.md) 的「第 1 层 用户态」+「2A 地址空间侧」+「第 3 层 MMU」。
> 🧭 **阅读前置**：先读 [00 总览](00-内存系统总览.md)；本篇是 [A2 缺页](A02-缺页与按需分页.md)、[A10 IOMMU/SVA](A10-IOMMU-SMMU与DMA.md)、[A12 大页/folios](A12-大页与页粒度.md) 的基础。

## 1. 定位：它在地图上的哪一格

这一篇把 [模块地图](01-连载规划与文章结构.md#1-模块层级与演进--篇幅地图) 里**横跨三层**的几个格子串起来讲：

- 用户态：进程地址空间布局、用户态分配器、`mmap`/`madvise` 等系统调用；
- 内核 2A：**VMA 管理**（"该映射成什么"）与**页表**（"当前实际映射成什么"）；
- 硬件：**MMU/TLB**（把虚拟地址翻成物理地址并缓存结果）。

它们合起来回答一个问题：**进程看到的"虚拟地址"，是怎么一步步变成 DRAM 上某个物理字节的。**

## 2. 它解决什么问题

直接让进程访问物理内存会带来三个无解的麻烦，虚拟内存正是为消除它们而生：

| 问题 | 没有虚拟内存 | 有了之后 |
|---|---|---|
| **隔离** | 任一进程可踩任意物理地址 | 每进程独立地址空间，越权访问触发异常 |
| **抽象** | 程序需自己安排物理布局、处理碎片 | 进程看到连续、私有、从 0 起的虚拟空间 |
| **过载使用 (overcommit)** | 物理内存用完即止 | 按需分页 + 回收/换出，可"承诺"超过物理量的内存 |

代价是每次访存都要做一次**地址翻译**——于是有了页表（翻译规则）与 MMU/TLB（翻译执行与缓存）。

## 3. 机制本体：当前是怎么做的

### 3.1 进程地址空间布局

进程的虚拟地址空间被切成若干区段：代码段（`.text`）/ 数据段（`.data`/`.bss`）/ 堆（`brk` 向上增长）/ 内存映射区（`mmap`，含共享库与匿名映射）/ 栈（向下增长）/ 顶部的内核区。

在 **ARM64** 上，这个空间被硬件一分为二，由两个基址寄存器分别管：

- **`TTBR0_EL1`** → 低半区（VA 高位全 0，`0x0000_…`），即**用户空间**；
- **`TTBR1_EL1`** → 高半区（VA 高位全 1，`0xFFFF_…`），即**内核空间**。

上半/下半各走哪个 TTBR 由硬件按 VA 高位自动选择，地址空间大小与翻译粒度由 `TCR_EL1` 配置。值得注意：**"TTBR0 给用户、TTBR1 给内核"是软件约定而非架构强制**，但所有主流 OS 都这么用——它的直接好处是**进程切换时只需换 TTBR0，内核映射（TTBR1）保持不变**。([Arm 寄存器手册](https://developer.arm.com/documentation/ddi0601/latest/AArch64-Registers/TTBR0-EL1--Translation-Table-Base-Register-0--EL1-))

### 3.2 VMA：地址空间"*应该*怎么映射"

内核用一组 **`vm_area_struct`（VMA）** 描述地址空间里每一段连续区间的属性：起止地址、权限（r/w/x）、后备对象（匿名 / 文件 + 偏移）、标志（共享/私有、是否可增长等）。VMA 是缺页处理的"裁判依据"——访问某地址时，先查有没有覆盖它的 VMA、权限是否允许，再决定补映射还是发 `SIGSEGV`。

VMA 的组织结构本身是个**长期演进点**：

- Linux **6.1** 起，VMA 的存储从红黑树 + 双向链表 + augmented tree 这套多结构，统一换成了 **maple tree**——一种对"非重叠区间"友好、缓存命中好、对 RCU 无锁读友好的 B 树变体。([Introducing the Maple Tree, LWN](https://lwn.net/Articles/845507/))
- Linux **6.4** 起引入 **per-VMA lock**：把原来进程级的 `mmap_lock` 细化为"每个 VMA 一把读写锁"。因为一次缺页只动到一个 VMA，没必要锁住整个地址空间。实测**缺页处理吞吐约翻倍**，尤其利好**多线程 App 启动**（Android 是直接受益者）。([Per-VMA locks, LWN](https://lwn.net/Articles/924572/)，[Linux 6.4, kernelnewbies](https://kernelnewbies.org/Linux_6.4))

> **VMA vs 页表，一句话**：VMA 说"这段地址*应该*映射成什么"，页表记"它*现在实际*映射成什么"。两者常不一致——按需分页下，VMA 早已建立，但页表项要等首次访问（缺页）才填上。

### 3.3 页表：地址空间"*当前实际*的映射"

页表是一棵以**虚拟地址分段做索引**的多级树，叶子（PTE）存"物理页帧号 + 权限/属性位"。ARM64 在 4KB 粒度、48 位 VA 的常见配置下是**四级**：

```
VA ─► PGD ─► PUD ─► PMD ─► PTE ─► 物理页帧 + 偏移
        (每级用 VA 的一段位做索引；PTE 里还有 AP 权限位、AF 访问位、共享性、内存类型等)
```

页大小直接决定 PTE 覆盖范围与页表深度：移动端常见 **4KB**，而 **Apple 芯片用 16KB 页**（页表更浅、TLB 覆盖更大）——这条线索在 [A12 大页与页粒度](A12-大页与页粒度.md) 展开。页表的**分配、填充、拆除**都在内核（`mm/` 下），但**格式由硬件定义、由 MMU 遍历**——这是典型的跨层协作。

### 3.4 MMU / TLB：硬件翻译与缓存

- **MMU** 按 `TTBRx_EL1` 指向的页表做**逐级遍历（page table walk）**，把 VA 翻成 PA，并按 PTE 的权限位做检查；
- **TLB** 缓存"VA→PA"的翻译结果，命中就免去一次多级遍历。TLB 是性能关键：一次 miss 可能触发数次访存去走页表；
- **ASID**（地址空间标识）给 TLB 项打进程标签，使进程切换时**不必整表刷新**，只换 TTBR0 + ASID；
- **TLB 失效（shootdown）**：当内核改了某 PTE（解除映射、改权限、回收换出），必须让对应 TLB 项失效，且要覆盖**所有可能缓存了它的 CPU**。x86 靠 IPI 中断逐核处理；ARM64 有**广播式 `TLBI` 指令 + DVM**，由硬件把失效广播到各核——这是回收/换出路径的固定开销。

### 3.5 用户态的出入口：分配器与系统调用

用户态不直接碰页表，而是通过两层进入：

- **用户态分配器**：`malloc`/`free` 先向内核要大块（`brk`/`mmap`），再在用户态切小块复用。**Android 自 11 起默认用 Scudo**（强化型分配器，替代 jemalloc；低内存设备仍用 jemalloc），iOS 用 libmalloc（nano/magazine 等）。([Scudo, AOSP](https://source.android.com/docs/security/test/scudo))
- **内存系统调用**：`mmap`/`munmap`（建/拆映射区）、`brk`（堆增长）、`mprotect`（改权限）、`mlock`（锁定不可换出）、以及关键的 **`madvise`**——它把"页的冷热/去留"意图直达回收侧（`MADV_DONTNEED` 解映射、`MADV_PAGEOUT/COLD/FREE` 触发回收，是 Android ART/Zygote 的常用手段，详见 [A4 回收](A04-回收总论.md)）。

## 4. 历史：为什么演变成今天这样（压缩）

地址翻译这套（多级页表 + MMU + TLB）几十年没大变；真正在动的是**地址空间的并发管理**：单把 `mmap_lock` 在重度多线程下成了瓶颈 → 先用 maple tree(6.1) 把数据结构换成 RCU 读友好的形态 → 再用 per-VMA lock(6.4) 把锁粒度降到单个 VMA。一条清晰的"为可扩展性持续重构"的脉络。

## 5. 平台差异（简）

| 维度 | Android (Linux) | iOS / Darwin | HarmonyOS |
|---|---|---|---|
| 用户态分配器 | **Scudo**（11+，低内存设备 jemalloc） | libmalloc | 待补 |
| 页大小 | 4KB（部分设备 16KB 迁移中） | **16KB** | 待补 |
| 地址空间并发 | maple tree + per-VMA lock（利好 App 启动） | XNU 自有 VM map | 待补 |

## 6. 趋势与未解问题（压缩）

- per-VMA lock 持续扩面：已扩展到 swap/userfault 等路径，并在向"所有构建默认可用"推进（[LWN](https://lwn.net/Articles/1070499/)）；
- **RCU 保护的 VMA 遍历**进一步降低缺页时的锁开销（[LWN](https://lwn.net/Articles/974392/)）；
- 页表本身正被 folios/大页（A12）与 IOMMU/SVA（A10）两个方向反复牵动——见下节。

## 7. 配合与依赖（本篇主体）

地址翻译这套机制本体稳定，**真正的复杂度在它与其他模块的耦合**。下面这些边对应 [模块地图](01-连载规划与文章结构.md#1-模块层级与演进--篇幅地图) 的依赖关系：

| 配合 | 方向与含义 | 去哪篇细看 |
|---|---|---|
| 缺页 → VMA | 访问某地址先查 VMA 校验权限/范围 | [A2](A02-缺页与按需分页.md) |
| 缺页 → 伙伴系统 → 页表 | 要物理页，回填并建立 PTE 映射 | [A2](A02-缺页与按需分页.md) / [A3](A03-物理页分配.md) |
| 回收 → 页表 | 换出/回收时**反向解除映射** + 刷 TLB | [A4](A04-回收总论.md) |
| `madvise` → VMA / 回收 | `DONTNEED` 拆映射；`PAGEOUT/COLD` 触发回收 | A4 |
| **页表 ↔ MMU/TLB** | 被遍历、被缓存、失效时广播刷新 | 本篇 §3.4 |
| **页表 ↔ IOMMU/SMMU 🔗** | **SVA**（共享虚拟寻址）让设备**直接共享进程页表**，GPU/NPU 用与 CPU 相同的 VA | **A10** |
| **页表 ↔ folios/大页 🔗** | 大页用更高层级的页表项（PMD 级）映射，提升 TLB 覆盖；shattering 又把它拆回小页 | **A12** |

> 后两条 🔗 正是本篇被定为"演进厚"的原因：页表本体不变，但 **IOMMU/SVA 让"页表"从 CPU 私有变成 CPU 与设备共享的资源**，**folios/16KB 让"一个 PTE 覆盖多大"成为持续调优的旋钮**——离开这两条耦合，就讲不清今天内存子系统为什么这么设计。

## 8. 实测 / 观测点

- `cat /proc/<pid>/maps`：看进程的 VMA 列表（每行 = 一个 VMA：地址区间、权限、后备文件）；
- `/proc/<pid>/smaps`：每个 VMA 的细分占用（Rss/Pss/匿名/共享等），度量细节见 [A13](A13-内存度量与排障.md)；
- `pmap -X <pid>`：等价的人读视图；
- `perf stat -e dTLB-load-misses,iTLB-load-misses`：观察 TLB 压力（大页/16KB 的收益就体现在这里）；
- ARM64 上 `TTBR0_EL1`/`TTBR1_EL1`/`TCR_EL1` 是页表与地址空间布局的硬件锚点。
