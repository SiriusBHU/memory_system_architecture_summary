# A05 · 冷热识别的演进（LRU → MGLRU → DAMON）

> **一句话定位**：回收的"瞄准镜"——内存吃紧时要先踢谁？这一篇讲内核如何给页排"冷热"，从教科书式的双链 LRU，到 refault/workingset 反馈，再到多代 LRU（MGLRU）与可编程的访问监控（DAMON）。

## 1. 定位：它在地图上的哪一格

回收（[A4](A04-回收总论.md)）回答的是"什么时候、回收多少、怎么把页写出去"；本篇只切其中一个动作：**在候选页里挑出"最该踢的"那些**。这件事在 [模块地图](01-连载规划与文章结构.md#1-模块层级与演进--篇幅地图) 上就是 `REC -->|选页| LRU` 这条边——回收是消费者，冷热识别是它的"瞄准镜"。

瞄得准不准，直接决定用户体感：踢错了页（把马上要用的踢走），它很快又被访问、触发缺页重新读回，这叫 **refault（再缺页）**；refault 多了就是 **thrashing（颠簸）**——CPU 全耗在搬页上，App 卡顿、被杀。所以"冷热识别"看似只是回收里的一小步，却是整条回收链的命门。

## 2. 它解决什么问题

回收必须做选择，而理想的选择标准是 **未来访问时间**——踢掉"最久才会再被用到"的页。但未来不可知，于是退而求其次，用 **过去的访问历史** 做预测，核心假设是 **时间局部性**：最近用过的，大概率还会再用。

这就需要一个能近似回答"哪些页最近没被碰过"的数据结构。理论上的 LRU（严格按最后访问时间全序排）代价太高——每次访存都要更新位置，无法接受。所以内核做的从来不是真 LRU，而是**用尽量低的开销，去逼近 LRU 的效果**。这条"逼近"之路上的每一次重构，就是本篇要讲的演进。

## 3. 机制本体：当前主线怎么做

主线内核里**同时存在两套实现**，编译期/运行期可切换：传统的双链 LRU（默认）、以及 MGLRU（[Linux 6.1 合入](https://www.phoronix.com/news/MGLRU-In-Linux-6.1)，需开启 `CONFIG_LRU_GEN`，主线默认不启用）。先讲沿用多年的双链法。

### 3.1 active / inactive 双链 + anon/file 分家

每个 [内存节点/memcg](A07-cgroup-memcg.md) 维护若干条 LRU 链表。关键是**两个维度的拆分**：

- **按冷热分**：`active`（活跃，近期被引用）与 `inactive`（不活跃，回收的首要候选）；
- **按后备分**：`anon`（匿名页，换出走 swap）与 `file`（文件页，干净的可直接丢、脏的需回写）。

四条链交叉组合：`active_anon`、`inactive_anon`、`active_file`、`inactive_file`。为什么 anon 与 file 要分家？因为两者回收代价不对称——干净文件页几乎零成本（丢了能从磁盘读回），匿名页必须先写到 swap。内核用 `swappiness` 旋钮在两类之间分配回收"火力"。

### 3.2 second-chance：用硬件 access bit 近似 LRU

页不是一访问就立刻移动位置（那太贵）。内核用的是 **second-chance（二次机会，CLOCK 算法变体）**：

- MMU 在访问页时**自动置位 PTE 的 access bit（AF）**（[A1 §3.4](A01-地址空间与虚实转换.md)）；
- 回收扫描 `inactive` 链尾时，若发现某页的 access bit 被置过，说明"它其实还在被用"，于是**清掉该位、给它一次机会**（晋升回 active 或留下），而不是立刻踢走；
- 真正被踢的，是扫到时 access bit 仍为 0 的页。

这样就用一个硬件位 + 偶尔扫描，近似出了"最近有没有被碰过"，避免了每次访存都更新链表。

### 3.3 refault / workingset：给"踢错"装上反馈

双链法有个致命盲区：**页一旦被踢出，内核就彻底忘了它**——下次它再被读回来，内核不知道"这是个被冤枉踢走的热页"，于是可能反复踢、反复读，陷入颠簸。

**workingset 检测**（Johannes Weiner，[Linux 3.15 合入](https://lkml.kernel.org/lkml/1375829050-12654-9-git-send-email-hannes@cmpxchg.org/)，针对 file 页）给这个盲区装上了反馈回路，思路很巧：

- 页被逐出 inactive 链时，不是无痕消失，而在 page cache 槽位留一个 **shadow entry（影子项）**，记下逐出当时的"逐出计数"快照；
- 该页 refault 回来时，用"当前逐出计数 − 快照"算出 **refault distance（再缺页距离）**——含义是"它不在内存的这段时间里，inactive 链又被挤掉了多少页"；
- 若这个距离 **小于 active 链长度**，说明"只要 inactive 链当时再长一点，它本不会被逐出"——即它属于**当前工作集**，于是把它**直接激活到 active 链**，并相应施压让 inactive 链变长。

一句话：**refault distance 让内核"事后复盘踢错的页"，把被冤枉的热页捞回来**。后来（约 Linux 5.8，Joonsoo Kim）这套检测**从 file 页扩展到了 anon 页**，匿名页也获得了同样的工作集保护。

> 这里和 [A6 换页](A06-压缩与换页.md)、[A8 压力与终止](A08-压力与低内存终止.md) 是一条线：refault 的统计正是 **PSI（压力失速信息）** 量化"内存颠簸"的数据来源——颠簸严重到一定程度，就该换页、甚至杀进程了。

## 4. 历史：为什么演变成今天这样 ← 本篇重心

把上面的机制按时间轴拉开，会看到一条清晰的脉络：**每一代都在解决上一代"扫描太贵 / 信息太少"的可扩展性瓶颈。**

**① 严格 LRU（理论起点）→ 太贵。** 每次访存都要把页移到链头，多核下还要抢锁，无法用于生产。

**② 双链 + CLOCK（second-chance）→ 用硬件位换近似。** 这是几十年的默认方案，便宜、够用。但它有两个先天缺陷：(a) **信息太粗**——只有 active/inactive 两档，"有多冷/多热"无法分级；(b) **扫描代价随内存增长**——回收时要扫物理页链表，几百 GB 内存的机器上，一次回收扫描可能动辄几十毫秒，且 access bit 是**逐页清**的，开销大。

**③ refault/workingset（3.15）→ 补上"踢出后失忆"。** 如 §3.3，给双链加了事后反馈，缓解颠簸。但它治标——纠错发生在"已经踢错并读回"之后，没改变扫描本身的低效。

**④ MGLRU（[6.1](https://lwn.net/Articles/894859/)）→ 从根上换数据结构与扫描方式。** Google 为 Chrome OS 与 Android 做的重写，针对的正是 ② 的两个缺陷：

- **多代（generations）取代两档**：不再是 active/inactive 二分，而是把页按"最后访问的代次"分进**多个 generation**（最冷的在 `min_gen_nr`，最热的在 `max_gen_nr`）。冷热分级细了，回收挑页更准。
- **aging 与 eviction："两只手的时钟"**：**aging hand（老化手）**周期性推进代次、把仍被访问的页提到最年轻代；**eviction hand（逐出手）**从最老的代里取页回收。两只手解耦，扫描节奏更可控。
- **走页表而非扫物理页**：MGLRU **遍历进程页表来清 access bit**（可批量清、含非叶级），相比"扫物理页链表逐个清"对缓存友好得多；为避免在稀疏页表上空走，还用 **Bloom filter** 跳过几乎没有活跃项的页表页。
- **PID 控制器做工作集保护**：MGLRU 引入一个**反馈控制器**，从"刚回收的页是否很快 refault 回来"中学习——发现踢错了，就调整老化/逐出的力度，避免误伤工作集。配套有 `min_ttl_ms` 旋钮：写入 N 表示"保护近 N 毫秒的工作集不被逐出"，文档建议 **`N=1000` 通常能消除因颠簸导致的难忍卡顿**（[内核文档](https://docs.kernel.org/admin-guide/mm/multigen_lru.html)）。

这一步是质变：**从"被动近似 + 事后纠错"转向"主动分级 + 反馈控制"**，且把扫描开销与内存规模解耦——这正是大内存机器与移动端共同的诉求。

**⑤ DAMON（[5.15](https://lwn.net/Articles/813108/)）→ 把"访问监控"本身独立成可编程子系统。** 见 §6.2，它代表的是另一个方向：不再把冷热识别**焊死在回收路径里**，而是抽出来做成通用框架。

## 5. 现状与平台差异

| 维度 | Android (Linux) | iOS / Darwin (XNU) | HarmonyOS |
|---|---|---|---|
| 冷热结构 | 双链 LRU；**MGLRU 在 14+ GKI 默认开** | XNU 自有页队列（active/inactive/speculative 等），非 Linux LRU | 待核实（基座相关） |
| 工作集保护 | refault/workingset + MGLRU 的 PID 反馈 | XNU 有自己的 reactivation 计数与 pageout 策略 | 待核实 |
| 主动回收 | `MADV_COLD/PAGEOUT`（[A1 §3.5](A01-地址空间与虚实转换.md)）、DAMON_RECLAIM（视内核版本） | 压缩内存 + jetsam（[A8](A08-压力与低内存终止.md)） | 待核实 |

**Android 的 MGLRU 采用**值得单独点出：MGLRU 在 **Android 14 的 GKI 内核（android14-5.15 与 android14-6.1）默认启用**，**Pixel 8 系列是首批默认开启的机型**；Google 公布的收益包括 App 启动时间下降、后台进程被杀减少、kswapd CPU 占用大幅下降（[报道](https://www.esper.io/blog/android-dessert-bites-22-linux-memory-management-38419756)）。运行期可用 `/sys/kernel/mm/lru_gen/enabled` 开关。

> iOS/XNU 用的是**另一套术语与结构**（page queues、compressor、jetsam），不要把它和 Linux 的 LRU/MGLRU 混为一谈——横向对比留待 [A14 平台对照](01-连载规划与文章结构.md#2-连载路线图阅读顺序--写作顺序)。

## 6. 趋势与未解问题 ← 本篇重心

### 6.1 MGLRU 的成熟与争议

MGLRU 合入后并非定论。其"走页表清 access bit"的做法在某些工作负载下被质疑开销与公平性，社区有[重新评估 MGLRU 的讨论](https://lwn.net/Articles/1060967/)。主线**默认仍是双链**、MGLRU 需显式开启——它在移动端（Android）已是默认，在服务器发行版则逐步铺开。未解问题：两套 LRU 长期并存的维护成本、以及 MGLRU 与 [memcg](A07-cgroup-memcg.md)、[分层内存](01-连载规划与文章结构.md#2-连载路线图阅读顺序--写作顺序)（tiering）等新场景的配合。

### 6.2 DAMON：把冷热识别"解耦成可编程框架"

如果说 MGLRU 是把回收路径里的 LRU 重写得更聪明，**DAMON 走的是另一条路：把"访问监控"从回收里抽出来，做成独立、低开销、可编程的子系统**。

- **DAMON（[Linux 5.15 合入](https://docs.kernel.org/mm/damon/index.html)）**：通过**基于区域的采样**监控访问模式——把地址空间切成若干 region，对每个 region 抽样查访问，再按访问情况自适应地拆分/合并 region。关键在于**开销与内存大小无关、可控**，因此能常驻生产环境。
- **DAMOS（DAMON-based Operation Schemes）**：在监控之上加"**条件 → 动作**"的规则——"对满足某访问特征（如某段时间没被访问）的区域，执行某操作"。动作可以是 `pageout`（换出）、`cold`/`hugepage` 提示等，**无需写代码、只配规则**即可做访问感知的系统操作。
- **DAMON_RECLAIM 与物理地址监控（[Linux 5.16](https://www.phoronix.com/news/DAMON-Reclamation-Linux-5.16)）**：在 DAMOS 之上实现的**主动回收**——自动找冷区域并提前回收，用于轻度内存压力；同时引入**物理地址空间监控**（不再局限单进程虚拟地址）。其设计意义是把"什么时候回收"从被动等水线，变成**可主动、可策略化**。

DAMON 的趋势价值在于它面向**新型内存层级**：CXL/分层内存（[A15 前沿](01-连载规划与文章结构.md#2-连载路线图阅读顺序--写作顺序)）下，"哪些页是冷的、该往慢层迁"正是 DAMON 擅长的——它已被用于 CXL 内存分层等真实场景。**一句话：MGLRU 让回收挑页更准，DAMON 让"冷热"成为可被任意策略消费的一等公民。**

### 6.3 per-memcg LRU：作用域问题

上面所有 LRU 链，**都是按 [memcg](A07-cgroup-memcg.md) 维度各维护一套的**——回收在某个 cgroup 内进行时，只动它自己的 LRU。这把"全局冷热"细化成"每个容器/应用的冷热"，是容器与 Android 应用隔离的基础。**这条线索在 [A7 cgroup/memcg](A07-cgroup-memcg.md) 展开**，本篇只点到为止。

## 8. 实测 / 观测点

- `cat /sys/kernel/mm/lru_gen/enabled`：看 MGLRU 是否启用（位掩码，bit0 为总开关）；`echo y > .../enabled` 运行期开启；
- `/sys/kernel/mm/lru_gen/min_ttl_ms`：MGLRU 工作集保护时长旋钮（建议值见 §4）；
- `cat /proc/meminfo` 中 `Active(anon)/Inactive(anon)/Active(file)/Inactive(file)`：四条双链的当量；
- `cat /proc/vmstat | grep workingset`：`workingset_refault*`、`workingset_activate*` 等，直接量化"踢错被捞回"的次数；
- `cat /proc/pressure/memory`：PSI 内存压力（refault/颠簸的高层表现，[A8](A08-压力与低内存终止.md)）；
- DAMON：用户态工具 [`damo`](https://github.com/awslabs/damo) 或 sysfs 接口 `/sys/kernel/mm/damon/` 配置监控与 DAMOS 方案。
