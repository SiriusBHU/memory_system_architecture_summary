# A03 · 物理页分配

> **一句话定位**：物理内存这块"砖"是怎么发出去的——伙伴系统按 2 的幂切/合页面、Zone 把物理内存按用途分段、水线决定何时唤醒回收，再到内核自己的 slab/vmalloc 把页拆成小对象。

> 🌡️ **演进分级**：**稳定薄**——伙伴系统、Zone、水线这套几十年大体未变，是教科书级机制。讲清**设计动机**即止，历史/趋势压到最小（唯一近年大事是 slab 家族收敛到 SLUB，§6 一句带过）。

## 1. 定位：它在地图上的哪一格

这一篇只管一件事：**当"上面"要一块物理内存时，"下面"怎么把它找出来、发出去。** 在 [模块地图](01-连载规划与文章结构.md#1-模块层级与演进--篇幅地图) 里它落在「2B 物理与回收侧」最底下两格：

- **页级分配**：伙伴系统（buddy allocator）——内核里**唯一**的物理页源头，所有人（缺页、page cache、DMA、甚至 slab 自己）都向它要页。
- **对象级分配**：slab/SLUB、kmalloc、vmalloc——把伙伴系统发来的整页**再切成小对象**给内核内部用。

它向上承接 [A02 缺页](A02-缺页与按需分页.md)（"要物理页，回填 PTE"），向下用水线机制触发 [A04 回收](A04-回收总论.md)（"页不够了，去回收"）。

## 2. 它解决什么问题

物理内存不是一片均质的、随便切的沙子，分配器要同时摆平三件事：

| 矛盾 | 没处理会怎样 | 伙伴系统/Zone 的应对 |
|---|---|---|
| **既要小块又要连续大块** | 频繁分/释后只剩零碎页，要 16 个连续页却拼不出 → **外碎片** | 按 2 的幂（order）管理，释放时**自动合并**相邻块 |
| **不同硬件对物理地址有约束** | 老式 DMA 设备只能寻址低 16MB，给它高地址页就用不了 | 把物理内存按用途切成 **Zone**，按约束从对应 Zone 取页 |
| **何时该腾地方** | 等内存彻底用完才回收 → 分配阻塞、抖动 | **水线**提前预警，后台 kswapd 边用边补 |

## 3. 机制本体：当前是怎么做的

### 3.1 伙伴系统：order、拆分与合并

物理页以 4KB（移动端常见粒度）为单位，编号成**页帧（page frame）**。伙伴系统按 **order**（阶）管理：order-`n` 表示一块 2ⁿ 个**连续**页帧。每个 order 维护一条空闲链表，最大到 `MAX_ORDER`（典型 1024 页 ≈ 4MB 一块）。

- **分配**：要 order-2（4 页）但该 order 空 → 向上找到 order-4 的空闲块，**逐级拆半**（split）：4→拆成两个 order-3，取一半…直到拆出 order-2，多出来的半块挂回各自 order 链表。
- **释放**：归还一块时，检查它的"**伙伴**"（地址只差一个 block 大小的相邻同 order 块）是否也空闲；若空闲就**合并**成更大的 order-`n+1`，并继续向上递归合并。这套"拆/合"是伙伴系统对抗外碎片的核心。

> 一句话记牢：**伙伴 = 地址对齐、大小相同、来自同一次拆分的那对相邻块**；合并只在伙伴双双空闲时发生。

### 3.2 迁移类型（migration type）：把"会动的"和"不会动的"分开放

光靠拆/合还不够——一个**不可移动**的内核小对象卡在大块中间，就永远合不出大页。于是伙伴系统在每个 order 上**再按迁移类型分独立空闲链表**：

- `MIGRATE_UNMOVABLE`：内核数据结构等，**不能搬**；
- `MIGRATE_MOVABLE`：用户匿名页、page cache 等，**可以搬**（改页表/folio 指向即可）；
- `MIGRATE_RECLAIMABLE`：可丢弃后重建的，如部分文件页。

把同类页**聚在相邻物理块**里，可移动页扎堆，将来 compaction（§3.6）才有连片的"可搬区"。某类用尽时会从其他类**偷块（steal）**，且优先从最大块偷以减少碎片。([Oracle Blog: Memory Compaction](https://blogs.oracle.com/linux/linux-memory-compaction-part-1)，[kernel-internals: Compaction](https://kernel-internals.org/mm/compaction/))

### 3.3 Zone：按硬件约束把物理内存切段

同一节点的物理内存按**寻址/用途约束**切成若干 Zone（实际是否存在取决于架构与内存大小）：

| Zone | 含义 | 谁用 |
|---|---|---|
| `ZONE_DMA` / `ZONE_DMA32` | 低物理地址区，给寻址受限的老式 DMA 设备 | 受限外设的连续内存 |
| `ZONE_NORMAL` | 内核可直接线性映射的常规内存 | 内核与一般分配的主战场 |
| `ZONE_MOVABLE` | **只放可迁移页**的人为划定区 | 利于热插拔/凑大页；CMA 也借此（见 [A09](A09-设备内存全景.md)） |
| `ZONE_HIGHMEM` | 32 位时代超出内核直接映射的高端内存 | 64 位下基本退场 |

分配请求带 **GFP 标志**（如 `GFP_KERNEL`/`GFP_ATOMIC`/`__GFP_DMA`）说明"从哪个 Zone 起、能不能睡眠、能不能触发回收"。分配器按 **zonelist** 顺序逐个 Zone 尝试，本节点不行再退到邻近节点（NUMA）。

### 3.4 页帧 `struct page`：每个物理页的"身份证"

内核为**每个**物理页帧配一个 `struct page`，记录它当下的身份：是匿名页还是文件页、引用计数、所在 LRU、是不是 slab/复合页的一部分等。它是连接物理页与上层（页表映射、LRU 回收、folio）的枢纽。`struct page` 本身常驻内存、数量与物理页数成正比，所以内核一直在压缩它的体积；近年的 **folio** 正是把"一组连续页"作为一个对象统一管理的方向（细节见 [A12 大页与页粒度]）。

### 3.5 水线（watermark）与分配路径：何时触发回收

每个 Zone 有三条水线，把"空闲页量"分成几个健康度区间：

| 水线 | 含义 | 触发什么 |
|---|---|---|
| **high** | 健康，无需动作 | kswapd 回收到此即停手 |
| **low** | 偏低预警 | 跌破即**唤醒 kswapd 后台回收**，目标补到 high |
| **min** | 危险下限 | 跌破后，**分配进程自己下场做直接回收（direct reclaim）**，期间阻塞 |

由此得出**分配路径**（`mm/page_alloc.c` 的核心逻辑）：

1. **快路径**：在水线之上，直接从伙伴系统对应 order/迁移类型链表取页，最快；
2. 取页使空闲跌破 **low** → **唤醒 kswapd**（[A04](A04-回收总论.md) 的后台回收线程）边走边补，分配本身不阻塞；
3. 若继续跌破 **min** → 进入**慢路径**：分配进程**直接回收**，必要时**触发 compaction（规整）**凑连片大块，仍失败则按 GFP 决定是重试还是返回 `NULL`，最坏唤起 **OOM killer**（见 [A04](A04-回收总论.md) / A08）。

`min` 与 `low` 之间的间距（即"留给 kswapd 抢救的缓冲"）可由 `/proc/sys/vm/watermark_scale_factor` 调。([kernel-internals: Page Reclaim](https://kernel-internals.org/mm/reclaim/)，[LWN: Tunable watermark](https://lwn.net/Articles/422291/))

### 3.6 外碎片与内存规整（compaction）

即便有迁移类型分组，长期运行后仍会外碎片化——**总空闲页够，却拼不出连续大块**（高 order 分配失败）。**compaction（规整）** 的做法：扫一个 Zone，把分散的**可移动页**搬到一端、空闲页腾到另一端，搬完就空出连片大块。它**只能搬 movable 页**——这正是 §3.2 迁移类型分组的回报。compaction 可由 kswapd 后台触发，也可在高 order 分配走慢路径时**直接 compaction**（同步、会带来延迟尖峰）。它对 THP/大页（A12）和 CMA/设备连续内存（[A09](A09-设备内存全景.md)）尤其关键。([kernel-internals: Compaction](https://kernel-internals.org/mm/compaction/))

### 3.7 内核对象分配器：slab/SLUB、kmalloc、vmalloc

伙伴系统最小也是**整页**，但内核内部大量是几十~几百字节的小对象（`task_struct`、`dentry`、inode…）。于是在伙伴系统之上叠一层**对象分配器**：

- **slab 层（现为 SLUB）**：向伙伴系统要整页，切成同尺寸对象的"缓存（cache）"，复用避免反复初始化、降低内碎片。每类对象一个 cache（可用 `slabinfo`/`/proc/slabinfo` 看）。
- **`kmalloc`**：通用小块分配，底层就是一组**按尺寸分级**的通用 slab cache，返回**物理连续**内存。
- **`vmalloc`**：要大块但**不要求物理连续**时用——它从伙伴系统拿若干**离散页**，在内核虚拟地址空间里**用页表拼成连续的虚拟区间**。代价是额外页表 + TLB 开销，故只在确实需要大而连续虚拟地址（如模块、大缓冲）时用。

> 收敛说明：内核历史上有 SLAB / SLUB / SLOB 三套 slab 实现。**SLOB 在 6.4 移除，SLAB 在 6.8 移除**（6.5 起已弃用），如今**只剩 SLUB**一套通用实现。([LWN: remove the SLAB allocator](https://lwn.net/Articles/951272/)，[Linux 6.8, kernelnewbies](https://kernelnewbies.org/Linux_6.8))

### 3.8 shrinker：回收侧反向回调（简述）

slab 里有些 cache（如目录项、inode 缓存）占用可观且**可被回收**。内存紧张时，回收子系统通过 **shrinker 接口**反向回调这些缓存"请释放一些"。本篇只点到为止——**shrinker 的注册、`count`/`scan` 两阶段语义、与 LRU 的配合属于回收侧**，细节见 [A04 回收总论](A04-回收总论.md)。

## 4. 历史（极简）

页级分配（伙伴 + Zone + 水线）是内存子系统里**最稳的地基**，多年只有参数化/可扩展性的小修。唯一值得记的近期变化是**对象分配器收敛**：三套 slab 实现裁到一套 SLUB（见 §3.7），让后续优化只需维护单一实现。

## 5. 平台差异（简）

伙伴系统/Zone/水线属 Linux 内核机制，**Android / HarmonyOS（基于 Linux 内核的形态）共享**这套；差异更多体现在**上层旋钮**与**设备连续内存**：Android 经 dma-buf heaps / CMA 向伙伴系统要可迁移大块（见 [A09](A09-设备内存全景.md)），并常调 `watermark_scale_factor`、`min_free_kbytes` 以适配低内存机型。iOS/XNU 的物理页管理是**独立实现**（非伙伴系统），术语与机制不可直接套用——见 [A14 · iOS/Darwin 平台对照](../platforms/A14-iOS-Darwin-内存实现.md)。

## 6. 配合与依赖

| 配合 | 方向与含义 | 去哪篇 |
|---|---|---|
| 缺页 → 伙伴系统 | 缺页要物理页，由伙伴系统供给并回填 PTE | [A02](A02-缺页与按需分页.md) |
| 水线 → 回收 | 跌破 low/min 唤醒 kswapd / 触发直接回收 | [A04](A04-回收总论.md) |
| 回收 → shrinker → slab | 内存紧张时反向回收可回收的 slab 缓存 | [A04](A04-回收总论.md) |
| compaction ↔ 大页 | 规整凑连片块以满足 THP/高 order 分配 | A12 |
| 伙伴系统 ↔ CMA / 设备内存 | 设备要连续大块，靠 movable 分组 + compaction | [A09](A09-设备内存全景.md) |

## 7. 实测 / 观测点

- `cat /proc/buddyinfo`：看每个 Zone 各 order 的空闲块数——**右侧（高 order）越空说明外碎片越重**；
- `cat /proc/zoneinfo`：看各 Zone 的 `min`/`low`/`high` 水线与当前空闲页；
- `cat /proc/slabinfo` 或 `slabtop`：看各 slab cache 的对象数/占用；
- `cat /proc/vmstat | grep compact`：看 compaction 的成功/失败/迁移计数；
- `/proc/sys/vm/` 下 `min_free_kbytes`、`watermark_scale_factor`、`compaction_proactiveness` 是相关旋钮。
