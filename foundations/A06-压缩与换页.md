# A06 · 压缩与换页（zram / zswap）

> **一句话定位**：当回收挑中一个匿名页要"换出"时，它去哪里？——本篇讲匿名页换出的**后端选择**，重点是两套"用 CPU 换内存"的压缩方案：在 RAM 内自成一块压缩块设备的 **zram**（移动端主力，不落盘），与把压缩缓存挡在真实 swap 设备前的 **zswap**。
>
> 📍 **对应总览**：[00 总览](00-内存系统总览.md) 的「2B 物理与回收侧 · 换页」格——它是回收路径的**出口之一**。
> 🧭 **阅读前置**：先读 [A04 回收总论](A04-回收总论.md)（谁触发换出）与 [A05 冷热识别的演进](A05-冷热识别的演进.md)（凭什么判定某页该被换出）；换入侧会触发 major 缺页，见 [A02 缺页与按需分页](A02-缺页与按需分页.md)。
> 🌡️ **演进分级**：**演进厚 ⚡**——机制本体（压缩 + 换出）思路稳定，但**写回（writeback）、多算法重压缩、memcg 感知的 shrinker、分层内存**几条线持续在动，**重点在 §4 历史 与 §6 趋势**。

## 1. 定位：它在地图上的哪一格

[A04 回收](A04-回收总论.md) 决定"该回收谁"，[A05](A05-冷热识别的演进.md) 决定"谁最冷"。但匿名页（堆、栈、`malloc` 出来的内存）没有文件后备——回收它不能像文件页那样"丢了再从磁盘读回"，必须先**把内容保存到某个换出后端**，否则进程数据就丢了。

这个"换出后端"就是本篇的主题。传统答案是**磁盘上的 swap 分区 / swapfile**；但在移动端，把热数据反复写进闪存既慢又伤寿命。于是出现了**压缩换页**：与其把页写到慢速存储，不如**就地压缩**，用 CPU 周期换回有效内存。本篇把这条"换出 → 压缩 → （可选）写回"的链路讲透。

## 2. 它解决什么问题（没有它会怎样）

匿名页换出的朴素方案是写磁盘 swap，三个痛点叠加：

| 痛点 | 直接写 swap 分区 | 压缩换页 |
|---|---|---|
| **延迟** | 换入是 major 缺页 + 一次磁盘 I/O，数毫秒级停顿 | 解压一页是微秒级，差几个数量级 |
| **闪存寿命** | 移动端 swap 落到 UFS/eMMC，频繁写放大磨损闪存 | zram 完全在 RAM 内，**默认不落盘** |
| **有效容量** | 受物理 swap 大小限制 | 匿名页压缩比常 2–3×，等于"扩容"了 RAM |

所以移动端的主流不是"有没有 swap"，而是"换出后端是不是压缩的、要不要落盘"。Android 与 iOS 给出的答案高度一致：**优先压缩在 RAM 内，尽量不碰闪存**。

## 3. 机制本体：当前是怎么做的

### 3.1 两条路线：zram vs zswap

这是本篇最容易混淆、也最该分清的一对（注意**区分术语**，不要把 Android 的 zram 与 iOS 的 compressed memory 混说）：

| 维度 | **zram** | **zswap** |
|---|---|---|
| 形态 | 一个 RAM 内的**压缩块设备** `/dev/zram0`，自身被 `mkswap`/`swapon` 当成 swap 用 | swap 子系统的**前置压缩缓存**（经 frontswap 钩子挂在真实 swap 设备前） |
| 是否需要磁盘 swap | **不需要**——它本身就是"swap" | **需要**——它只是缓存，背后必须有已配置的物理 swap 设备 |
| 不可压缩页怎么办 | 只能**原样存**在 RAM 里（没省下内存） | 可直接**拒收**，让 swap 子系统把原页写真实磁盘 |
| 默认是否落盘 | **不落盘**（除非显式开 writeback） | 池满时按 **LRU 把压缩页逐出到背后的磁盘 swap** |
| 典型场景 | **移动端 / 无磁盘 swap 的嵌入式**（Android 默认） | 桌面 / 服务器：已有磁盘 swap，想减少 swap I/O |

一句话记忆：**zram 是"压缩后就是终点"的 RAM swap；zswap 是"压缩后挡在磁盘 swap 前面"的缓存。** 二者历史上同源（见 §4），但定位正交，通常不同时启用同一段 swap。([In-kernel memory compression, LWN](https://lwn.net/Articles/545244/))

### 3.2 zram：RAM 里的压缩块设备

zram 创建 `/dev/zram<id>` 设备，**写进去的页在内存里被压缩存储**，绕过块 I/O 子系统。它当 swap 用时，回收路径把冷匿名页"写"给它，zram 压缩后把压缩块交给 **zsmalloc** 管理（见 §3.4）。([zram 文档, kernel.org](https://docs.kernel.org/admin-guide/blockdev/zram.html))

支持的压缩算法包括 **lzo / lzo-rle / lz4 / zstd** 等——这是一条**速度↔压缩比**的权衡轴：lz4 快、压缩比中等（移动端常用）；zstd 压缩比高、CPU 开销大。Android 设备上的 zram 普遍以 **lz4** 为默认。

### 3.3 zswap：磁盘 swap 前的压缩缓存

zswap 用 frontswap 钩子，在页**即将被换出到磁盘 swap 时**先尝试压缩进一个动态 RAM 池；命中则避免一次磁盘写。它**必须有一个真实 swap 设备兜底**：

- 不可压缩 / 压缩收益不足 → 直接放行，让原页写磁盘 swap；
- **池满** → 按 **LRU 把最冷的压缩页逐出（writeback）到背后的磁盘 swap**，腾出 RAM。([zswap 文档, kernel.org](https://docs.kernel.org/admin-guide/mm/zswap.html))

zswap 还有一个**可选的 shrinker，默认关闭**：开启后，在内存压力下主动把池里的冷压缩页写回磁盘 swap、回收这部分 RAM，避免冷数据长期占着内存（详见 §6）。([zswap 文档, kernel.org](https://docs.kernel.org/admin-guide/mm/zswap.html))

### 3.4 zsmalloc：压缩页的分配器

压缩后的页是**变长**的（一页可能压成几百字节），普通 slab 分配器存这种碎块很浪费。**zsmalloc** 专为此设计：按大小分级、能把多个压缩对象紧凑塞进物理页帧，**密度高**。它不返回直接可寻址的指针，而是返回一个 **handle，访问前需先 map**——这换来了高密度，代价是**不利于并发逐出**（碎片让"挑一页搬走"变难）。([zswap 文档, kernel.org](https://docs.kernel.org/admin-guide/mm/zswap.html)、[LWN](https://lwn.net/Articles/545244/))

> 对照：早期 zswap 也可用 **zbud**——每个页帧最多塞 2 个压缩页，密度低但**逐出可预测**。"密度 vs 可逐出性"是压缩存储后端的核心张力。

## 4. 历史：为什么演变成今天这样 ←（演进重点）

**一条"在 RAM 里压缩"的主线，分叉成两支。** 早期 Linux 有过 `compcache`/`ramzswap` 之类的探索：把页压缩进 RAM 当 swap 用。最终沉淀为两条进入主线的路径：

1. **zswap 先入（v3.11）**——作为磁盘 swap 的**前置压缩缓存**，靠 frontswap 钩子；初期用 zbud 做后端，强调"压缩收益不够就放行到真实 swap"。([zswap, LWN](https://lwn.net/Articles/549740/))
2. **zram 随后转正（v3.14）**——连同 **zsmalloc** 一起**从 staging 毕业**进入主线，定位为**独立的 RAM 压缩块设备**，不依赖任何磁盘 swap。([In-kernel memory compression, LWN](https://lwn.net/Articles/545244/))

历史上曾有"把 zram 并进 zswap"的提案，但因二者定位正交（一个是终点、一个是缓存）未被采纳——这恰恰说明**两者不是替代关系，而是面向不同场景**。

zram 自身的关键演进，是**逐步具备"落盘"能力**——但这是**可选项、且专为冷/不可压缩页设计**，与"默认不落盘"并不矛盾：

- **`CONFIG_ZRAM_WRITEBACK`**：允许把**空闲（idle）页**或**不可压缩页**写回一个后备块设备（需在设置 `disksize` 前先配置 backing device），让这些"压了也没用 / 久未访问"的页离开宝贵 RAM。([zram 文档, kernel.org](https://docs.kernel.org/admin-guide/blockdev/zram.html))
- **`CONFIG_ZRAM_MULTI_COMP`**：支持 **1 个主算法 + 至多 3 个次算法**，对**冷的压缩页用更强的算法重压缩（recompress）**，进一步压低 zsmalloc 的内存占用——典型组合是"热页用 lz4 求快、冷页用 zstd 求省"。([zram 文档, kernel.org](https://docs.kernel.org/admin-guide/blockdev/zram.html))

可以看到，zram 的演进方向不是"变成磁盘 swap"，而是**在 RAM 内做更细的冷热分层**（重压缩）+ **给真正没价值的页一个可选出口**（writeback）。这与 §6 的分层内存趋势同向。

## 5. 现状与平台差异

| 维度 | **Android（Linux）** | **iOS / Darwin（XNU）** | **HarmonyOS** |
|---|---|---|---|
| 换出后端 | **zram**（默认启用，作为 swap）；冷页可选 writeback 到 swapfile | **compressed memory**（XNU VM compressor），iOS 上无传统磁盘 swap | 待核实 |
| 压缩算法 | 常用 **lz4**（部分配置 lzo-rle / zstd） | **WKdm + LZ4**（XNU 内核压缩器） | 待核实 |
| 落盘策略 | **默认不落盘**（护闪存）；writeback 仅给 idle / 不可压缩页 | iOS 历史上**只压缩不落盘**；macOS 有 compressor + swap 后端 | 待核实 |
| 术语 | zram / swappiness | "compressed memory"（Instruments 里显示的 *swap* 实为压缩内存） | 待核实 |

**Android 默认 zram、且默认不向闪存落盘**，核心动机就是 §2 的闪存寿命：UFS/eMMC 的写入次数有限，把热数据反复换进换出会显著加剧写放大与磨损，而压缩进 RAM 既快又零写放大。([zram 文档, kernel.org](https://docs.kernel.org/admin-guide/blockdev/zram.html))

> **术语警示**：iOS 的 *compressed memory* 是 XNU VM compressor 的产物，机制上与 Linux 的 zram **不是同一套代码、不是同一套术语**——它用 WKdm/LZ4、由 XNU 的 pager 框架驱动。横向类比"思路相同"可以，但**不要把 zram 的 sysfs 旋钮套到 iOS 上**。

## 6. 趋势与未解问题 ←（演进重点）

压缩换页正从"单层压缩"走向"**分层内存（tiered memory）的一环**"。几条活跃方向：

- **memcg 感知、压力驱动的 zswap shrinker**：传统 zswap 只在"撞到用户设的上限"时才收缩，导致冷压缩页长期占 RAM；2023 年起的工作把 zswap 的 LRU 拆成 **per-memcg / per-NUMA**，让 shrinker 能在**内存压力下做工作负载相关的写回**，并避免一个 memcg 误写另一个 memcg 的页，实测内核 CPU 时间下降约 **14%–35%**。([workload-specific zswap writeback, LWN](https://lwn.net/Articles/953429/))
- **与 MGLRU 的代际信息打通**：让压缩池的 LRU 复用 [A05](A05-冷热识别的演进.md) 里 MGLRU 的"页代际"信息，使"该写回哪页"判断更准——把冷热识别与压缩后端连成一条线。([LWN](https://lwn.net/Articles/953429/))
- **zram 内部分层**：`MULTI_COMP` 重压缩 + writeback，本质是在"未压 RAM → lz4 压缩 → zstd 重压 → 后备设备"之间做**多级冷热分层**。
- **指向 A15 的大趋势**：CXL、far memory、硬件压缩把"内存"从单一 DRAM 摊成**有速度差的多层**。压缩换页是这条谱系里**最靠近 CPU 的一层**——"压缩内存"与"CXL 远端内存 / 硬件压缩内存"在框架上正在被统一为"**有代价差的换出/换入目标**"。详见 A15 前沿·先进内存。

**未解问题**：压缩比与解压延迟的权衡随负载漂移（没有放之四海皆准的算法选择）；zsmalloc 高密度与并发逐出的矛盾；以及 shrinker 何时该把 RAM 让给别人——这些都还在调。

## 8. 实测 / 观测点

- `zramctl` / `/sys/block/zram0/`：看 zram 设备的 `disksize`、`mm_stat`（原始 vs 压缩字节、压缩比）、`bd_stat`（writeback 后备设备统计）、`comp_algorithm`（当前/可选算法）；
- `/sys/block/zram0/recomp_algorithm` 与 `recompress`：多算法重压缩的旋钮（`MULTI_COMP`）；
- `/sys/module/zswap/parameters/`：`enabled`、`compressor`、`zpool`（zsmalloc/zbud）、`shrinker_enabled` 等运行时参数；
- `/proc/vmstat` 里 `pswpin`/`pswpout`、`zswpin`/`zswpout`/`zswpwb`：换入/换出与 zswap 写回计数；
- Android：`dumpsys meminfo` 看 SwapPss、`cat /proc/swaps` 确认 zram 作为 swap 挂载（度量细节见 [A13](A13-内存度量与排障.md)）；
- iOS / macOS：Instruments 的 VM Tracker / "Compressed Memory" 列——注意它显示的 *swap* 实为压缩内存。
