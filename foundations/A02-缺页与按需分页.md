# A02 · 缺页与按需分页

> **一句话定位**：把"虚拟地址*应该*映射成什么"（VMA）真正落实成"*现在实际*映射到哪个物理页"（PTE）的那一刻——访问一个尚未建立映射的地址，CPU 抛出缺页异常，内核按需补上物理页并重执行指令。

>

> 📍 **对应总览**：[00 总览](00-内存系统总览.md) 的「2A 地址空间侧」的「缺页处理 / 按需分页·COW」两格，是把 A01 的 VMA/页表"激活"的运行时路径。

> 🧭 **阅读前置**：先读 [A01 地址空间与虚实转换](A01-地址空间与虚实转换.md)（VMA、页表、PTE）；本篇向下接 [A03 物理页分配](A03-物理页分配.md)（缺页向伙伴系统要页）、[A11 page cache 与回写](A11-page-cache与回写.md)（文件页命中/读入）、[A06 压缩与换页](A06-压缩与换页.md)（swap-in 触发 major 缺页）。

> 🌡️ **演进分级**：机制本体**稳定（中等）**——COW、按需分页、零页是教科书级的几十年老设计，近年变化主要在*并发*（per-VMA lock 下的缺页路径，见 A01）而非*语义*。本篇适度展开，压缩历史与趋势。

## 1. 定位：它在地图上的哪一格

A01 讲清了"静态结构"：VMA 描述一段地址*应该*怎么映射，页表记录它*当前实际*映射到哪。本篇讲的是**两者之间那条动态的桥**——程序访问一个 VMA 已覆盖、但页表项还空着的地址时，硬件与内核怎么协作把这一格"点亮"。这条路径几乎是整个内存子系统最热的代码：每一次首次访问、每一次 fork 后的写、每一次 swap-in，都从这里进。

## 2. 它解决什么问题：按需分页（demand paging）

如果进程一 `mmap` 100MB 就立刻分配 100MB 物理页，绝大多数永远用不到的页会被白白占用——这正是 A01 提到的 **overcommit** 不可能成立的世界。

**按需分页**的策略是：`mmap`/`brk` 只建立 VMA（登记"这段地址将来该映射成什么"），**不碰物理页、不填页表**。直到 CPU 真正访问某个页，触发缺页，内核才临时找一个物理页填进去。于是"承诺的地址空间"可以远大于物理内存，代价是把分配的时机推迟到了缺页那一刻——这就是本篇要拆解的路径。

## 3. 缺页的两类：minor vs major

内核按"是否需要访问后备存储（磁盘/swap）"把缺页分成两档，分别累加到 `task_struct` 的 `min_flt` / `maj_flt`，记账逻辑在 `mm/memory.c` 的 `mm_account_fault()`：

| | **minor fault（次缺页 / 软缺页）** | **major fault（主缺页 / 硬缺页）** |
|---|---|---|
| 定义 | 所需数据**已在物理内存里**，只是当前进程没有指向它的有效 PTE | 所需数据**不在内存**，必须从磁盘/swap 读入才能建立映射 |
| 典型场景 | 首次访问匿名页（给零页）；文件页已在 page cache 命中；fork 后的 COW 写 | 文件首次读入（page cache 未命中、走 FS 读盘）；被换出的页 swap-in |
| 代价 | 只需分配/绑定物理页 + 建 PTE，纳秒~微秒级 | 含一次 I/O，毫秒级，是延迟敏感场景的痛点 |

> 区分的实益：性能排障时 `maj_flt` 高通常意味着内存吃紧、在频繁 swap 或冷启动读盘；`min_flt` 高只是正常的按需分页与 COW，未必是问题。([minor/major 定义与 `mm_account_fault`](https://johnnysswlab.com/debugging-performance-issues-in-kernel-space-minor-fault-and-major-faults/))

## 4. 缺页处理的完整路径

一次缺页从 CPU 异常进入，到指令被重新执行，大致是这样一条链（ARM64 上从同步异常向量进 `do_page_fault`，核心逻辑在 `mm/memory.c` 的 `handle_mm_fault` → `handle_pte_fault`）：

```
① CPU 访存,MMU 走页表发现 PTE 无效/权限不符 → 抛缺页异常,陷入内核
② 按 fault 地址查 VMA(A01):
     · 找不到覆盖它的 VMA            → SIGSEGV(非法访问)
     · 找到了但权限不符(如写只读页)  → 看是否 COW;否则 SIGSEGV
③ 分两种后备来源补页:
     · 匿名页:向伙伴系统(A03)要一个物理页
              —— 首次"读"通常给只读零页;首次"写"才要真页(见 §5/§6)
     · 文件页:先查 page cache(A11)
              —— 命中 → minor fault,直接映射
              —— 未命中 → 经 VFS/文件系统读盘填入 cache → major fault
④ 建立 PTE:把物理页帧号 + 权限位写进页表(A01 的页表被"点亮")
⑤ 返回用户态,重新执行那条触发缺页的指令 —— 这次 MMU 翻译成功
```

第 ② 步是 VMA 作为"裁判"的体现（A01 §3.2）；第 ③ 步把本篇接到了 A03（要页）与 A11（page cache）；第 ⑤ 步的"重执行"是缺页对用户态透明的关键——程序完全不知道刚才发生过一次陷入。

## 5. 零页（zero page）

匿名内存有一条专门的快路径。进程刚 `mmap` 一块匿名区并**首次去读**时，内容本就该是全 0，没必要立刻分配真页——内核直接把一张**全局共享的只读零页**（`empty_zero_page`/`ZERO_PAGE`）映射给这个 PTE，并把 PTE 设为**写保护**。于是成千上万个"已分配但还没写过"的匿名页可以共用同一张物理页，几乎零成本。

只有当进程**真正去写**这个页时，写保护触发又一次缺页，内核此时才向伙伴系统要一张真页、清零、改成可写并替换映射。这就把"分配真页"的时机进一步推迟到了"首次写"。([零页只读映射 + 首读给零页、首写才分配](https://www.chudov.com/tmp/LinuxVM/html/understand/node35.html))

## 6. 写时复制（COW）

COW 是按需分页在**共享场景**下的延伸：让多个进程先共享同一份只读物理页，谁要改谁才复制。

**fork 的实现**：传统 `fork()` 若把父进程整个地址空间逐页深拷贝，既慢又浪费（子进程常立刻 `exec` 把这些页全丢掉）。实际做法是——fork 时**不复制数据页**，只复制页表项，并把父子双方对应的 PTE 都标成**只读**，同时 VMA 仍保持可写。之后任一方去写，写只读页触发缺页；内核发现"**VMA 可写但 PTE 只读**"这一矛盾，就识别出这是 COW 页，于是分配新页、拷贝内容、把写者的 PTE 改成可写私有副本（`do_wp_page`/`wp_page_copy`），另一方继续用原页。([COW 识别：VMA 可写而 PTE 只读；fork 时双方 PTE 置只读](https://www.chudov.com/tmp/LinuxVM/html/understand/node35.html))

**Android Zygote 是 COW 的旗舰用例**：系统启动时，Zygote 进程预加载约数千个框架类与公共资源（`preloaded-classes`），把 ART 运行时状态"冻结"在内存里。每启动一个 App，都是从 Zygote **fork** 出来——子进程通过 COW **共享这份只读的框架与类数据**，只有被实际写脏的页才各自复制。这既让 App 冷启动跳过了重复的类加载，又让全机多个 App 共享同一份框架内存。([Android Zygote 预加载 + fork 共享，写时复制](https://www.codecentric.de/en/knowledge-hub/blog/android-zygote-boot-process))

> 注意区分平台术语：Android 的"Zygote fork + COW"是 Linux fork/COW 语义的直接应用；iOS/Darwin 无 Zygote 等价物，App 由 `launchd`/`posix_spawn` 起新进程，其共享主要靠 dyld 共享缓存（dyld shared cache）映射只读系统库，机制不同，勿混为一谈。

## 7. fault-around：用一次缺页顺手映射邻居

对文件页有一个减少 minor fault 的优化。访问某文件页缺页时，它周围的页**往往已经在 page cache 里**（比如顺序读、刚被预读进来），却还各自缺着 PTE——每个都单独走一次缺页太亏。**fault-around** 让缺页处理顺手把 fault 地址附近、且已在 page cache 命中的若干页**一并建立映射**，通过 `vm_operations_struct` 的 `->map_pages()` 回调完成，且要求**不阻塞**（页不在 cache 就跳过，绝不为此去读盘）。

默认窗口 `fault_around_bytes` 对应 **16 个页**（`FAULT_AROUND_PAGES = 16`，order 4）。引入它的实测里，16GB 文件顺序访问的 minor fault 从 5.03 亿降到 140 万——量级的削减。([fault-around 引入 patch 与实测数据，LWN](https://lwn.net/Articles/588802/))

## 8. userfaultfd：把缺页处理权交给用户态（简述）

通常缺页只有内核能处理。**`userfaultfd`** 是一个系统调用，让用户态程序注册一段地址区间、并在该区间发生缺页时收到通知，由用户态决定"用什么内容填这一页"（通过 `ioctl` 把页填入或置零），相当于 `PROT_NONE + SIGSEGV` 那套老技巧的正规、更高效替代。

主要用途：

- **虚拟机热迁移的 postcopy**（QEMU/KVM 的招牌场景）——目的端先让 VM 跑起来，访问到尚未传过来的内存页时由 userfaultfd 通知迁移处理器按需拉取；
- **用户态自定义按需分页**——以页粒度精确控制、避免持有 VMA 级别的重型锁；
- **写保护追踪（WP 模式）**——在用户态做脏页跟踪等。

它的价值在于把"缺页该怎么补"这个决策点从内核挪到用户态，同时保持页粒度精度。([userfaultfd 用途，内核 admin-guide 文档](https://www.kernel.org/doc/html/latest/admin-guide/mm/userfaultfd.html)、[User-space page fault handling, LWN](https://lwn.net/Articles/550555/))

## 10. 实测 / 观测点

- `/usr/bin/time -v <cmd>`：输出 "Major (requiring I/O) page faults" 与 "Minor" 计数；
- `ps -o min_flt,maj_flt -p <pid>`、`cat /proc/<pid>/stat`（第 10、12 字段为 minflt/majflt）；
- `perf stat -e minor-faults,major-faults <cmd>`：直接计数两类缺页；
- `vmstat`、`sar -B`：看系统级 `fault/s`、`majflt/s`；
- fault-around 旋钮：`/sys/kernel/debug/fault_around_bytes`（调窗口大小）。
