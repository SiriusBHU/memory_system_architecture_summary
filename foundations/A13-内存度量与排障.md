# A13 · 内存度量与排障

> **一句话定位**：把前面各篇讲过的机制"看见"——一个进程到底占了多少内存、哪些是它私有的、哪些和别人分摊、设备内存藏在何处，以及当 RAM 紧张时怎么读懂这些数字定位元凶。
>
> 📍 **对应总览**：[00 总览](00-内存系统总览.md) 的「第 1 层 用户态 · 内存度量」与「2B 物理与回收侧 · 共享去重」，是横切所有层的"观测面"。
> 🧭 **阅读前置**：先读 [A01 地址空间与虚实转换](A01-地址空间与虚实转换.md)（smaps 来自 VMA）、[A09 设备内存全景](A09-设备内存全景.md)（dmabuf/GPU 计量）、[A07 cgroup/memcg](A07-cgroup-memcg.md)（memory.stat）；冷热相关字段见 [A05 冷热识别的演进](A05-冷热识别的演进.md)。
> 🌡️ **演进分级**：**中等（实战向）**——机制本身稳定，价值在"怎么看到、怎么读懂、怎么排障"。本篇是把全连载落到 `/proc` 与 `dumpsys` 字段上的"对账篇"。

## 1. 定位：它在地图上的哪一格

度量不是独立的子系统，而是贯穿四层的**观测面**。它回答一个反复出现的实战问题：**"这台设备/这个进程的内存到底花在哪了，谁能背锅，谁不能。"** 本篇的组织顺序也照此推进：先把单进程的四个口径（VSS/RSS/PSS/USS）讲清，再下钻到 `/proc` 的原始字段，然后扩到系统级（`meminfo`/`memcg`）与设备内存，最后落到 Android 的成品工具与一节"共享与去重"。

## 2. 它解决什么问题

朴素的"进程占多少内存"没有唯一答案——因为共享。一份 libc 被几百个进程映射，一个 dma-buf 被相机进程分配却被显示合成器持有。把共享页**重复计入每个进程**会让总和远超物理内存；只算私有又会漏掉真实开销。于是衍生出四个口径，各自回答不同的问题：

| 口径 | 全称 | 含义 | 回答什么 |
|---|---|---|---|
| **VSS** | Virtual Set Size | 进程虚拟地址空间总大小（= VMA 总和） | "它*申请/映射*了多大地址范围"——大量虚高，几乎无排障价值 |
| **RSS** | Resident Set Size | 当前驻留物理内存的部分 | "它*占了多少物理页*"——但共享页被每个进程**全额重复计入** |
| **PSS** | Proportional Set Size | 私有页全算 + 共享页**按分摊**算 | "把它**公平**地摊一份是多少"——所有进程 PSS 之和≈物理用量 |
| **USS** | Unique Set Size | 仅进程**私有**（Private）的部分 | "杀掉它能**立刻**回收多少"——私有脏页是硬开销 |

**PSS 如何按共享分摊**是本篇的核心定义。内核 `proc` 文档给的算法是：**一个映射的 PSS = 私有页全额 + 每个共享页"除以共享它的进程数"**。文档原例：某进程有 1000 个独占页、另有 1000 个与另一进程共享的页，则它的 PSS = 1000 + 1000/2 = **1500 页**。([kernel.org · filesystems/proc](https://www.kernel.org/doc/html/latest/filesystems/proc.html)) Perfetto 的说法更直观：把一个 4KiB 页映射进 4 个进程，每个进程的 PSS 只增加 1KiB。([perfetto.dev · 内存调试](https://perfetto.dev/docs/case-studies/memory)) PSS 之所以是 Android 内存核算的"主货币"，正因它**可加**——把全系统进程的 PSS 相加大致不会重复计数。

> **关系速记**：USS ⊆ PSS ⊆ RSS ⊆ VSS。USS 是 PSS 里"分摊系数=1"（无人共享）的那部分。

## 3. 机制本体：从 /proc 原始字段读起

PSS/USS 不是凭空算的，数据源就是 [A01](A01-地址空间与虚实转换.md) 讲的 **VMA**。`/proc/<pid>/maps` 给出每个 VMA 的地址区间与后备文件；`/proc/<pid>/smaps` 在每个 VMA 下再细分占用：

- **Rss**：该映射当前驻留 RAM 的量；
- **Pss**：该映射的**比例份额**（私有全算 + 共享按进程数分摊），见 §2；
- **Shared_Clean / Shared_Dirty**：被多进程映射、未改 / 已改的页；
- **Private_Clean / Private_Dirty**：仅本进程映射、未改 / 已改的页——**两者之和≈USS**，其中 **Private_Dirty 是无法被直接丢弃的硬开销**（干净页可被内核回收后从后备文件重读，脏页必须先写出）；
- **Referenced**：标记为最近被访问的量（冷热判定的痕迹，见 [A05](A05-冷热识别的演进.md)）；
- **Anonymous**：不属于任何文件的匿名内存；
- **Swap / SwapPss**：已换出到 swap 的匿名量 / 其**比例换出份额**。

（以上字段定义均据 [kernel.org · filesystems/proc](https://www.kernel.org/doc/html/latest/filesystems/proc.html)。）

逐 VMA 的 smaps 信息量大但慢。内核提供 **`/proc/<pid>/smaps_rollup`**：把全部映射的同名字段**汇总成一份总计**，并额外给出 **Pss_Anon / Pss_File / Pss_Shmem**——按"匿名 / 文件 / 共享内存"三类拆分的 PSS。要一个进程的总 PSS/USS，读 `smaps_rollup` 一次即可，比遍历 smaps 高效得多。([kernel.org · filesystems/proc](https://www.kernel.org/doc/html/latest/filesystems/proc.html))

## 4. 系统级口径：/proc/meminfo 与 memcg

单进程看完，往上看整机。**`/proc/meminfo`** 是系统级总账，排障常看：

- **MemTotal / MemFree**：可用 RAM 总量 / 完全空闲量；**MemAvailable**：估算"不触发换出能给新应用多少"——比 MemFree 更贴近"还剩多少可用"，紧张判断优先看它；
- **Buffers / Cached**：块设备缓冲 / 文件页缓存（page cache，见 A11）；**SwapCached**：已在 swap 又仍在内存的页；
- **Active / Inactive**：LRU 两条链的规模（[A05](A05-冷热识别的演进.md) 的冷热基础）；
- **AnonPages / Mapped / Shmem**：匿名页 / 映射文件页 / 共享（tmpfs）内存；
- **Slab / SReclaimable / SUnreclaim / KReclaimable**：内核对象缓存总量，及其中"压力下可回收 / 不可回收"的拆分——**SUnreclaim 偏高常指向内核侧泄漏**；
- **Dirty / Writeback**：待回写 / 正回写的脏页量。

（字段含义据 [kernel.org · filesystems/proc](https://www.kernel.org/doc/html/latest/filesystems/proc.html)。注意这是 **Linux `/proc` 口径**，与下文 Android `dumpsys` 的分类体系是两套术语，勿混用。）

把范围收窄到一个 cgroup，则读 **`memory.stat`**（cgroup v2，见 [A07](A07-cgroup-memcg.md)）：`anon` / `file` / `kernel_stack` / `slab` / `shmem` 等按 memcg 作用域统计。Android 把应用归入各自 cgroup 后，`memory.stat` 是"这一组进程"的精确账本，且新内核为**导出的 dma-buf 增加了 memcg 计数**，使设备内存也能落到 cgroup 维度。([LWN · Track exported dma-buffers with memcg](https://lwn.net/Articles/919548/))

## 5. 设备内存的计量：最容易"丢账"的地方

GPU、相机、显示用的缓冲常以 **dma-buf**（[A09](A09-设备内存全景.md)）形式存在。它的麻烦在于**所有权与使用权分离**：分配者（如 gralloc）和持有者（如合成器）可能是不同进程，按进程 RSS/PSS 去找它**容易漏算或重复算**。配套观测手段：

- **`/sys/kernel/dmabuf/buffers/`**：Android 12 引入的 dma-buf sysfs 统计（`CONFIG_DMABUF_SYSFS_STATS`），每个 buffer 以 inode 编号给出大小与导出者，`libdmabufinfo` 据此提供**按导出者 / 按 buffer** 的视图。([AOSP · Implement DMABUF and GPU memory accounting in Android 12](https://source.android.com/docs/core/graphics/implement-dma-buf-gpu-mem))（注：该 sysfs 接口因每 buffer 建删文件的开销已被标注 DEPRECATED，演进方向转向 fdinfo / BPF，属**待持续跟踪**项。）
- **GPU 私有内存**：驱动私有分配未必出现在通用计数里，需厂商 GPU 的专有节点；
- **unreclaimable**：pinned / 设备占用的页**不可回收**，在压力下直接挤占可用量（[A08](A08-压力与低内存终止.md) 的低内存终止常因此被触发），排障时要把它与可回收的 page cache 区分开。

## 6. Android 成品工具：把上面的字段读成人话

`/proc` 字段是底料，Android 把它们加工成几个常用命令：

- **`dumpsys meminfo <pkg|pid>`**：最常用。以 **Pss Total / Private Dirty / Private Clean** 为列，把进程内存按**用途**归类——典型条目有 **Native Heap**（C/C++，含图形/物理引擎缓冲）、**Dalvik Heap**（ART 托管的 Java/Kotlin 对象）、**Dalvik Other**（JIT 代码缓存、GC 开销、DEX 解压等运行时杂项）、**Stack**（各线程调用栈）、**.so/.dex/.apk mmap**（映射的库、字节码、资源）、以及 **Ashmem / Gfx dev / EGL mtrack / GL mtrack** 等设备与图形类。([perfetto.dev · 内存调试](https://perfetto.dev/docs/case-studies/memory)) 注意这是 Android 的**用途视角分类**，和 `/proc/meminfo` 的内核视角是两套口径。
- **`procrank`**：按 VSS/RSS/PSS/USS **四列并排**列出全系统进程，一眼看出"谁最占"与"杀谁最划算（看 USS）"；
- **`showmap <pid>`**：等价于把 smaps 按映射逐行展开，定位是哪个 `.so` / 哪段匿名占了大头；
- **`pmap -X <pid>`**：Linux 通用的人读版 maps/smaps，桌面与设备皆可。

排障路径通常是：`dumpsys meminfo` 看分类锁定可疑大头 → `showmap`/`smaps` 定位具体映射 → 结合 `meminfo`/`memcg` 判断是单进程问题还是整机压力。

## 7. 共享与去重（轻量一节）

度量绕不开"共享"，因为共享既压低了 PSS，也是节省 RAM 的手段。两个机制值得知道：

- **KSM（Kernel Samepage Merging，相同页合并）**：内核线程 `ksmd` 周期扫描**已登记**的匿名内存区，发现**内容完全相同**的页就合并成**一份写保护页**，任一方改写时再 COW 拆开。仅作用于匿名页，不动文件页。开启方式：进程对内存区调用 `madvise(MADV_MERGEABLE)`；Linux **6.4** 起新增 **`prctl(PR_SET_MEMORY_MERGE, 1)`**，可**进程级一键开启**且 fork 时继承——解决了 GC 语言拿不到 madvise 的痛点。统计在 `/sys/kernel/mm/ksm/`：`pages_shared`（合并后保留的页数）、`pages_sharing`（因此省下的额外引用数）、`general_profit`（净收益）。它**最初为 KVM 而生**（让多虚拟机共享相同内容以塞下更多 VM），现已通用化。([kernel.org · admin-guide/mm/ksm](https://www.kernel.org/doc/html/latest/admin-guide/mm/ksm.html)、[LWN · Process-level KSM control](https://lwn.net/Articles/928510/)) 代价是 `ksmd` 的扫描 CPU 开销，移动端是否启用取决于厂商权衡。
- **memfd（共享匿名内存）**：`memfd_create` 创建一个**匿名、有 fd 句柄**的内存对象，可经 fd 传递在进程间**共享同一份匿名页**——常作为 ashmem 的现代替代与跨进程缓冲载体。它是"主动共享"，KSM 是"事后去重"，两者互补：前者在度量上表现为 Shared_*，后者把原本 Private 的相同页转成共享从而压低 PSS。

## 8. 实测 / 观测点速查

| 想看什么 | Linux 通用 | Android 专有 |
|---|---|---|
| 进程虚拟布局 / 映射 | `/proc/<pid>/maps`、`pmap <pid>` | `showmap <pid>` |
| 进程 PSS/USS/私有脏页 | `/proc/<pid>/smaps`、`smaps_rollup` | `dumpsys meminfo <pid>` |
| 全系统进程横比 | — | `procrank`（VSS/RSS/PSS/USS 四列） |
| 整机内存总账 | `/proc/meminfo` | `dumpsys meminfo`（无参，全局） |
| cgroup 维度 | `memory.stat`（[A07](A07-cgroup-memcg.md)） | 应用 cgroup 的 `memory.stat` |
| dma-buf / GPU | `/sys/kernel/dmabuf/buffers/` | `libdmabufinfo`、厂商 GPU 节点 |
| 去重收益 | `/sys/kernel/mm/ksm/` | （同左，视内核配置） |
