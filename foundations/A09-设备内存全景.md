# A09 · 设备内存全景（GPU / dma-buf / CMA）

> **一句话定位**：终端内存里那个不在 `malloc`、不在 page cache、却动辄占掉一两 GB 的"隐形大户"——GPU 显存、图形缓冲、相机/视频/显示链路所用的设备内存，以及把它们**跨设备零拷贝共享**（dma-buf）、**保证物理连续**（CMA）、**钉住不回收**（pinned）的那一整套机制。
>

## 1. 定位：它在地图上的哪一格

前面几篇讲的都是"**CPU 能直接寻址、能被回收子系统管理**"的内存：匿名页、文件页、slab。本篇讲的是另一类——**为外设（GPU、Display、Camera、Codec、NPU）准备的内存**。它的特殊性在于三点：

- **使用者不是 CPU 而是 DMA 设备**：内容要被 GPU 着色器、显示控制器扫描、视频编解码器读写；
- **常常要求物理连续 / 特定地址约束**：不带 IOMMU 的外设只认连续物理地址（见 [A10](A10-IOMMU-SMMU与DMA.md)）；
- **要在多个设备、多个进程之间传递而不复制**：相机出帧 → GPU 合成 → 显示送显，全程同一块缓冲。

这三点决定了它**不能走普通的页分配与 LRU 回收路径**，于是长出了 dma-buf / CMA / gralloc / 各类 heap 这一套平行机制。

## 2. 它解决什么问题（没有它会怎样）

设想没有统一的设备缓冲框架：相机驱动分配一块内存、GPU 要用就得**拷贝**一份到自己的格式、显示控制器要扫描再**拷**一份。一帧 4K 图像约 33MB，60fps 下每多一次拷贝就是约 2GB/s 的内存带宽与功耗——移动端根本扛不住。

设备内存框架要同时解决四件事：

| 需求 | 机制 |
|---|---|
| **跨设备/跨进程零拷贝共享** | dma-buf（用一个 fd 代表缓冲，谁拿到 fd 谁就能映射同一块物理内存） |
| **给"只认连续物理地址"的外设供页** | CMA（连续内存分配器）/ carveout |
| **图形缓冲的语义协商**（格式、对齐、用途、tiling） | gralloc（Android 的图形内存分配 HAL） |
| **受保护内容隔离**（DRM / 安全视频） | secure / restricted heap（CPU 不可读的受保护内存） |

## 3. 机制本体：当前是怎么做的

### 3.1 dma-buf：缓冲共享的通用语

**dma-buf** 是内核里"**在多个设备驱动与子系统间共享缓冲（供 DMA 访问）、并同步异步硬件访问**"的框架（[内核 dma-buf 文档](https://www.kernel.org/doc/html/latest/driver-api/dma-buf.html)）。核心角色两个：

- **exporter（导出方）**：真正分配并管理底层存储的驱动，实现 `struct dma_buf_ops`，决定缓冲"实际落在哪、怎么分配"；
- **importer / buffer-user（导入方）**：消费这块缓冲的驱动，"**不需要关心缓冲怎么分配、在哪**"，只需拿到描述其内存的 scatterlist。

关键设计：**dma-buf 以一个文件描述符（fd）暴露给用户态**，从而可以"**在进程、子系统、设备之间传递**"——这正是零拷贝共享的实现根基。导入方通过 `dma_buf_attach()` → `dma_buf_map_attachment()` 把缓冲映射进自己的（设备）地址空间，用完 `dma_buf_unmap_attachment()` → `dma_buf_detach()`。设备间的读写顺序由 **`dma_resv` 里的隐式 fence（栅栏）** 协调，"在异步硬件操作完成时发出信号"。CPU 若要直接访问，必须用 `dma_buf_begin_cpu_access()` / `dma_buf_end_cpu_access()` 括起来，以处理 cache 一致性。

> **一句话**：dma-buf 把"一块设备缓冲"抽象成"一个 fd + 一份 ops"，让相机、GPU、显示、编解码器拿着同一个 fd 各自映射、零拷贝接力。

### 3.2 gralloc：图形缓冲的"分配协商"

dma-buf 是底层管道，**gralloc** 是 Android 图形栈在其上的分配层。SurfaceFlinger / 应用通过 gralloc HAL 申请缓冲，传入宽高、像素格式、用途位（`USAGE_GPU_RENDER` / `USAGE_VIDEO_ENCODER` / `USAGE_PROTECTED` …）；gralloc 据此决定对齐、tiling、从哪个 heap 分配，最终返回一个 dma-buf fd 包装进 `GraphicBuffer`。换言之 gralloc 回答"**这块图形缓冲该长什么样**"，dma-buf 回答"**这块内存怎么被共享**"。

### 3.3 CMA：给"要连续物理内存"的外设供页

很多移动外设没有（或不用）IOMMU，只能 DMA 到**物理连续**的区域。但系统跑久了物理内存碎片化——即使空闲几个 GB，也凑不出连续 8MB。**CMA（Contiguous Memory Allocator）** 的解法是（[A deep dive into CMA, LWN](https://lwn.net/Articles/486301/)）：

- **启动时预留一块区域**，保证连续空间存在；
- 平时**只允许可迁移页（movable，如 page cache）借用**这块区域，不浪费；
- 当设备真要连续缓冲时，**把借用的可迁移页迁移走**，腾出连续区段交给设备。

这条"预留 + 借用 + 按需迁移"正是它与伙伴系统的接口——CMA 区即伙伴系统里的 `MIGRATE_CMA` 迁移类型（见 [A03](A03-物理页分配.md)）。它也解释了一个常见现象：**CMA 区的页平时计入可回收/可迁移，设备一申请就被夺回**，从内存压力角度看是一种"随时会蒸发"的可用内存。

### 3.4 heap 家族：从同一框架分出不同来源

dma-buf heaps 框架（见 §4）把"缓冲从哪来"做成可枚举的 heap，每个 heap 是 `/dev/dma_heap/<name>` 下一个独立字符设备：

- **system heap**（`/dev/dma_heap/system`）：分配**虚拟连续、可缓存**的缓冲，底层 `alloc_page()`，最常用；
- **cma heap**：分配**物理连续、可缓存**的缓冲，底层走 CMA。其节点名因平台而异——内核文档指出 6.17 之前它"可能叫 `reserved`、`linux,cma` 或 `default-pool`"（[dma-buf-heaps UAPI 文档](https://docs.kernel.org/userspace-api/dma-buf-heaps.html)）；
- **secure / restricted heap**：从**受保护内存**分配，供 DRM、安全视频路径使用——**CPU 侧不可读映射**，内容只在受信执行环境/硬件解码器间流转。厂商常以 SoC 专有 heap 形式提供（如 `/dev/dma_heap/<vendor-name>`）。

### 3.5 pinned：为什么设备内存"赖着不走"

设备正在 DMA 的缓冲**必须 pinned（钉住）**——物理页不能被移动、不能被换出，否则 DMA 会写到错误地址。后果是：**这些页不进普通 LRU、不被 kswapd/直接回收选中、不能换页到 zram**。在 `/proc/meminfo` 里它们体现为 **`Unreclaimable`/`KReclaimable` 之外的不可回收占用**，是回收子系统"看得见却动不了"的内存。这条性质是 §7 压力耦合的根源。

## 4. 历史：从 ION 到 dma-buf heaps（演进重点）

**ION** 是 Android 早年（约 2011 起）自带的统一设备内存分配器（[The Android ION memory allocator, LWN](https://lwn.net/Articles/480055/)），统一了各家厂商五花八门的 carveout/pmem 方案，按 heap（system / carveout / cma / 各厂商私有）分配并导出 dma-buf。但 ION 长期是**out-of-tree、UAPI 不稳定**的——各厂商可自定义 flags 与 heap ID，导致"没法写一套通用测试框架"，也带来安全面（曾被安全研究界点名，如 Black Hat Asia 2022 的 ExplosION 议题）。

社区的做法是**把 ION 里通用、干净的部分"去暂存（destage）"进上游主线**，成为 **dma-buf heaps** 框架（[DMA-BUF Heaps (destaging ION), LWN](https://lwn.net/Articles/792016/)）。它**于内核 5.6 进入主线**，初始带 system heap 与 cma heap 两种。相对 ION 的三大改进：

1. **安全更细**：每个 heap 是独立字符设备，可用 **sepolicy 对单个 heap 单独做访问控制**（ION 做不到）；
2. **ABI 稳定**：IOCTL 接口在上游维护，**ABI stable**；
3. **UAPI 标准化**：提供"**well-defined UAPI**"，消除 ION 自定义 flag/heap-ID 的碎片化。

Android 侧的切换节点很明确（[Transition from ION to DMA-BUF heaps, AOSP](https://source.android.com/docs/core/architecture/kernel/dma-buf-heaps)）：**Android 12 的 GKI 2.0 用 dma-buf heaps 取代 ION**，`android12-5.10` 内核分支于 **2021-03-01 关闭 `CONFIG_ION`**。为平滑迁移，AOSP 提供抽象库 **`libdmabufheap`**：先按名字找 dma-buf heap，找不到再回退到等价的 ION heap。需要留意的现实是：**仍有海量存量设备（停留在更老内核/版本）继续用 ION**——这是 Android 设备碎片化的典型缩影。

> **演进脉络一句话**：厂商各自为政的 carveout → ION 统一但 out-of-tree、UAPI 不稳 → 把干净部分 destage 进主线成 dma-buf heaps（内核 5.6）→ Android 12/GKI 2.0 强制切换、关闭 CONFIG_ION。

## 5. 现状与平台差异

| 维度 | Android | iOS / Darwin | HarmonyOS |
|---|---|---|---|
| 设备缓冲框架 | dma-buf heaps（12+ 强制）/ 存量 ION | IOSurface（系统级共享 surface） | 沿用 Linux dma-buf（OpenHarmony 内核态）（待核实具体形态） |
| 图形分配层 | gralloc HAL → dma-buf fd | IOSurface + Metal heaps | 待补 |
| 连续内存 | CMA / 厂商 carveout | 统一内存架构（UMA），CPU/GPU 共享物理内存 | CMA（待核实） |
| 受保护内容 | secure/restricted heap + TEE | Secure Enclave / 受保护 IOSurface | TEE（待补） |

> **术语区分**：Android 说 **dma-buf / gralloc / heap**，Apple 说 **IOSurface / unified memory**，二者解决的问题域重叠但**不要混用术语**。Apple 的 UMA 让 CPU 与 GPU 共享同一物理内存池，从根上减少了"显存独立、需拷贝"的问题，但也意味着 GPU 占用直接挤压系统可用内存——挤压逻辑与 Android 一致。

## 6. 趋势与未解问题（演进重点）

- **GPU cgroup 控制器**：长期痛点是 GPU/dmabuf 内存**不受 memcg 约束**（见 [A07]），一个进程的 GPU 占用无法被 cgroup 限额。社区在推 **GPU/DRM cgroup controller**，目标把设备内存纳入 cgroup 计量与限制（RFC 阶段，[lkml RFC](https://lkml.iu.edu/2203.1/02691.html)，待核实落地版本）。
- **受限/受保护 dma-buf 的标准化**：secure heap 长期是厂商私有，社区在尝试**上游统一的 restricted/secure dma-buf** 语义，让 DRM 路径不再依赖各家私货（待核实进展）。
- **更细的归属计量**：dmabuf 难以归属到"是谁的内存"——同一 fd 被多进程共享时怎么算？Android 12 的 per-exporter/per-buffer 统计是一步（见 §8），但**跨进程共享缓冲的公平计量**仍未解。
- **与 IOMMU/SVA 的合流**：带 SMMU 的平台越来越多，设备可经 IOMMU 用非连续物理页（弱化对 CMA 的依赖），甚至经 **SVA 直接共享进程页表**（见 [A10](A10-IOMMU-SMMU与DMA.md)）——长期看 CMA 的角色会向"无 IOMMU 的低端外设"收缩。

## 7. 配合与依赖（跨层耦合）

| 配合 | 方向与含义 | 去哪篇细看 |
|---|---|---|
| **CMA ← 伙伴系统** | CMA 区是伙伴系统的 `MIGRATE_CMA` 类型，借用/夺回可迁移页 | [A03](A03-物理页分配.md) |
| **设备地址转换 ↔ IOMMU/SMMU 🔗** | 有 IOMMU 时设备可用非连续物理页；scatterlist 经 SMMU 映射成设备虚拟地址 | [A10](A10-IOMMU-SMMU与DMA.md) |
| **pinned 挤压 → 压力与终止 🔗** | 设备内存不可回收，吃掉的物理内存挤压可回收池，**抬高水线压力、提前触发 LMKD/Jetsam** | [A08](A08-压力与低内存终止.md) |
| **计量归属 → 度量排障** | dmabuf/GPU 内存的归属与重复计数是排障难点 | [A13](A13-内存度量与排障.md) |
| **受保护 heap ↔ TEE** | secure heap 内容只在受信环境流转，CPU 侧无可读映射 | 本篇 §3.4 |

> **这条最要命**：设备内存是 pinned 的（§3.5），它越多，**留给回收子系统腾挪的可回收内存就越少**。于是在低内存机型上，一个吃显存的游戏 + 几路相机/视频缓冲，能在系统"看起来还有内存"时就把可回收池压干，让 PSI 飙升、LMKD 提前杀后台——这正是 [00 总览](00-内存系统总览.md) 里 `设备内存 → 压力与终止` 那条边的实质（详见 [A08](A08-压力与低内存终止.md)）。

## 8. 实测 / 观测点

设备内存的**计量难题**本身值得单列：它**不在 `/proc/meminfo` 的常规可回收项里**，普通工具看不全。Android 12 起补了一套（[Implement DMABUF and GPU memory accounting, AOSP](https://source.android.com/docs/core/graphics/implement-dma-buf-gpu-mem)）：

- **`/sys/kernel/dmabuf/buffers`**：Android 12 新增的 dma-buf sysfs 统计，`libdmabufinfo` 解析它得到**按 exporter / 按 buffer** 的占用；导出驱动须正确设置 `dma_buf_export_info.exp_name`；
- **`dmabuf_dump`**：可用 `-b` 输出按 exporter 的统计；
- **GPU 私有内存**：通过 `gpu_mem` tracepoint / 厂商 GPU 驱动的 sysfs 节点暴露（节点名厂商相关，待核实）；
- **`dumpsys meminfo`**：Android 12 把图形内存的会计从 `MemtrackType::GRAPHICS` 计入 totalPss，改为用 **`totalExportedDmabuf`** 计入 `lostRAM`——已映射的 dmabuf 顶替 Memtrack HAL 的 Graphics 类，未映射的 dmabuf 计入内核内存（避免重复计数与厂商口径不一）；
- **`/proc/meminfo`**：关注 pinned/设备内存导致的 **不可回收占用**（与 `KReclaimable`/`SReclaimable` 对照看差额），这是判断"内存去哪了"的关键缺口（度量方法论见 [A13](A13-内存度量与排障.md)）。
