## 1. 定位：它在地图上的哪一格

前面 A01–A13 讲的内存系统，**默认了一个隐含前提：主存就是一整块 DRAM，访问代价均匀**。回收（[A04](../foundations/A04-回收总论.md)）挑出冷页后，去向只有两个——文件页丢回存储、匿名页压进 [zram/swap](../foundations/A06-压缩与换页.md)。本篇要打破的就是这个前提：

> **当"内存"本身变成多层、每层速度/容量/成本不同时，回收不再是"踢出去"，而是"往下挪一层"——这就是分层内存（tiered memory）。**

把谱系画清楚（越往上越快/小/贵）：

```
   寄存器 / L1-L3 缓存（SRAM）
   ├─ 本地 DRAM / LPDDR              ← 传统"主存"
   ├─ 压缩内存（zram / 压缩器）       ← A06：用 CPU 换容量，最靠近 CPU 的"换出层"
   ├─ CXL 扩展内存（本机、稍慢的 DRAM）← 本篇 §3
   ├─ CXL 池化 / 远端内存（跨主机）   ← 本篇 §4，数据中心
   └─ 慢存储（PMEM / UFS / SSD swap） ← 最慢的换出后端
```

[A06 §6 的伏笔](../foundations/A06-压缩与换页.md) 在这里收束：压缩内存与"CXL/远端/硬件压缩内存"在框架上正被统一成"**有代价差的换出/换入目标**"。本篇就是这条谱系的总论。

需先说明**范围与立场**：CXL、远端内存、硬件压缩**几乎全部面向数据中心**，终端目前基本接触不到 CXL 硬件——但"分层 + 冷热迁移 + 压缩"的**思路**正反哺移动端（见 §6）。故本篇前半是数据中心视角的事实梳理，末节回到终端。

## 2. 它解决什么问题（没有它会怎样）

驱动这一切的是一对**长期失衡**：

| 失衡 | 现象 | 分层内存的回应 |
|---|---|---|
| **容量墙** | CPU 核数与内存带宽/容量需求增长快，但单插槽 DRAM 的引脚/通道数有上限，"加内存"越来越贵且受物理约束 | 用 CXL 在 PCIe 物理层上**扩展**内存，绕开 DIMM 插槽数限制 |
| **资源搁浅（stranded memory）** | 数据中心里某台机器内存用满、邻居机器内存大量闲置却无法借用，整体利用率低、TCO 高 | 内存**池化/解耦**，让一池内存按需分给多台主机 |

传统答案只有"往机器里多插 DIMM"或"换出到慢存储"。前者撞容量墙、后者慢几个数量级。分层内存提供了**中间层**：比 DRAM 慢一点、但比落盘快得多，且能跨设备甚至跨主机共享。代价是引入了**非均匀访问**——同一段物理内存，放在本地 DRAM 还是 CXL 上，延迟可能差 2–3 倍（综述量级，具体随实现差异很大，待核实）。于是"哪些页该放哪层、什么时候迁"成了新的核心问题——这正好是 [A05 DAMON](../foundations/A05-冷热识别的演进.md) 擅长的。

## 3. 机制本体：四块拼图

### 3.1 分层内存 tiered memory：内核怎么把"多层"管起来

内核把不同速度的内存**抽象成不同的 NUMA 节点**，再把节点归入有高低之分的"内存层（memory tier）"。这套机制是渐进落地的：

- **回收时降级（demote-on-reclaim，Linux 5.15）**：在持久内存（PMEM）等场景下，回收冷 DRAM 页时**不再直接换出到 swap，而是先"降级"到更慢的 NUMA 节点**；只有更慢的层也满了才落盘。由 `/sys/kernel/mm/numa/demotion_enabled` 开关控制。([Linux 5.15 demote during reclaim, Phoronix](https://www.phoronix.com/news/Linux-5.15-Demote-During-Reclai))
- **显式内存层与 demotion 子系统（Linux 6.1，`mm/memory_tier.c`）**：早期"层级"是按 NUMA 节点间距离**隐式**推出来的，CPU 热插拔还会扰动层级；6.1 起改为**显式分层**——所有内存节点默认在一个中间层（abstract distance 200），**设备驱动可把自己的节点上调或下调**（如 GPU 内存上调、PMEM/CXL 下调），CPU 增减不再影响层级。([mm/demotion: Memory tiers and demotion, LWN](https://lwn.net/Articles/901205/))
- **NUMA-balancing 式的提升（promotion）**：复用 NUMA balancing 机制——周期性给页做**访问保护**、由缺页采样判断"这页在慢层却很热"，再**提升（promote）**回快层；`NUMA_BALANCING_MEMORY_TIERING` 让它独立于 demotion 开关工作。([NUMA balancing for memory tiering, LWN](https://lwn.net/Articles/849095/))

一句话：**降级走回收路径（冷页往下沉），提升走 NUMA balancing（热页往上浮），两条流共同维持"热数据在快层"。** 但这套基于 NUMA balancing 的提升有个公认短板——它靠缺页采样的**滑动窗口反应慢**，热页"等着被提升"的这段时间性能在持续衰减（[NUMA rebalancing on tiered-memory systems, LWN](https://lwn.net/Articles/893024/)）。这正是 §6 里 DAMON 方案试图解决的。

后续还在动的：**本机直连内存的分层**改进（[Better support for locally-attached-memory tiering, LWN](https://lwn.net/Articles/974126/)）、**按 cgroup 控制提升/降级的公平性**（[memory tiering fairness by per-cgroup control, LWN](https://lwn.net/Articles/991190/)）、以及面向带宽而非仅容量的**加权交织（weighted interleaving）**（[Weighted interleaving for memory tiering, LWN](https://lwn.net/Articles/948037/)）。

### 3.2 CXL（Compute Express Link）：内存语义的物理层扩展

**CXL 是什么**：一套**架在 PCIe 物理层之上的缓存一致互连协议**（CXL 自 1.1 起复用 PCIe 5.0、后续复用 PCIe 6.0 电气层）。它把一条 PCIe 通道"升级"成可以跑内存语义的链路，含三个子协议：

- **CXL.io**：等价于 PCIe 事务，做发现、配置、DMA；
- **CXL.cache**：让设备一致地缓存主机内存；
- **CXL.mem**：让主机以**加载/存储（load/store）内存语义**访问设备上的内存——这是"内存扩展"的关键。

按用到哪些子协议，设备分三类：**Type-1**（带 cache 的加速器，如 NIC）、**Type-2**（带本地内存的加速器，如 GPU，io+cache+mem 全用）、**Type-3**（**纯内存扩展器**，主要用 CXL.io + CXL.mem）。本篇关心的"内存扩展/池化"几乎都是 **Type-3 设备**。([CXL device types, logic-fruit](https://www.logic-fruit.com/blog/cxl/compute-express-link-cxl-device-types/)、[What is CXL 3.0, Synopsys](https://www.synopsys.com/blogs/chip-design/what-is-compute-express-link-3.html))

**规范版本与能力演进**（核实自 CXL Consortium 公告与二手汇总，日期以官方公告为准）：

| 版本 | 发布 | 关键能力 |
|---|---|---|
| 1.0 / 1.1 | 2019 | 基础一致互连；Type-1/2/3 设备；点对点内存扩展 |
| 2.0 | 2020-11 | **单层交换（switching）**、**内存池化（pooling）**、热插拔、安全（IDE） |
| 3.0 | 2022-08 | 复用 PCIe 6.0；**多级交换**、**内存共享（sharing，多主机一致访问同一内存）**、Fabric/GFAM、多头设备 |
| 3.1 | 2023-11 | Fabric 扩展、可信执行环境相关增强 |
| 3.2 / 4.0 | 2024–2025 | 持续扩展（具体能力**待核实**，新于本篇主要参考时点） |

> **扩展 vs 池化 vs 共享，别混**：**扩展（expansion）**= 给单台主机加一块稍慢的内存（1.1 即可）；**池化（pooling）**= 一池内存按需切给多台主机、各用各的（2.0）；**共享（sharing）**= 多台主机**一致地访问同一份**内存（3.0）。能力是逐级递增的。([CXL 3.0 white paper](https://computeexpresslink.org/wp-content/uploads/2023/12/CXL_3.0_white-paper_FINAL.pdf))

**内核怎么看 CXL 内存**：Linux 把 CXL.mem 暴露的内存放进一个**没有 CPU 的独立 NUMA 节点**（CPU-less node），按性能等级（由固件 SRAT/HMAT 描述）归类——性能相近的多设备可并进同一节点。驱动栈以 `cxl_pci` / `cxl_*` 为核心。这样一来，CXL 内存就**自然嵌进 §3.1 的分层框架**：它通常被放进**比本地 DRAM 低的层**，作为 demotion 的目标。([Memory-management changes for CXL, LWN](https://lwn.net/Articles/931416/)、[CXL: Management and tiering, LWN](https://lwn.net/Articles/894598/)、[CXL Reclaim, kernel.org](https://docs.kernel.org/driver-api/cxl/allocation/reclaim.html))

**现状要清醒**：CXL 今天能稳定工作的主要是**内存扩展**（开机即用、混合 IMC+CXL 内存）；**运行时池化、跨主机共享仍属"增强能力"而非成熟形态**，落地受设备生态、软件成熟度与商业竞争多重不确定性影响——这点 LWN 的讨论里有明确的怀疑声音（[Uses for CXL memory, LWN](https://lwn.net/Articles/1017599/)）。**这是综述判断，请按时间衰减看待。**

### 3.3 远端内存 / 内存解耦（far memory / disaggregation）

把 §3.2 的"池化"推到极致，就是**内存解耦**：计算与内存在数据中心里**物理分离**，机器按需从一池远端内存借用容量。学界已研究多年，方案谱系大致两类：

- **应用集成式**：如 **AIFM**（Application-Integrated Far Memory，OSDI 2020），让应用通过特定数据结构/运行时主动管理本地与远端内存的搬运，开销低但需改应用。([AIFM, OSDI 2020](https://dl.acm.org/doi/10.5555/3488766.3488784))
- **透明换页式 + 可靠性**：如 **Hydra**（FAST 2022）等，把远端内存当成对应用透明的"另一层 swap"，并解决远端内存的**可用性/容错**（远端节点宕了不能丢数据）。
- **综述**：[*Memory Disaggregation: Advances and Open Challenges*（SIGOPS OSR 2023）](https://arxiv.org/pdf/2305.03943) 对这条线做了系统梳理，并指出未解难题集中在**延迟、容错、缓存一致性与网络栈**。

历史上远端内存靠 **RDMA over 网络** 实现，延迟和一致性都吃力；CXL 3.0 的**缓存一致共享**被寄望成为更干净的硬件底座——但二者关系仍在演化（RDMA 软件栈成熟、CXL Fabric 硬件尚早），**谁主导尚无定论（综述观点）**。

### 3.4 硬件压缩 / 近内存计算（near-memory computing）

[A06](../foundations/A06-压缩与换页.md) 的压缩换页是**用 CPU 周期换容量**；下一步自然的想法是**把压缩/计算从 CPU 下沉到内存侧**，省掉数据来回搬运的能耗与带宽：

- **近内存 / 存内计算（PNM / PIM）**：在内存模组或 DRAM bank 旁放轻量计算单元。代表如三星 **HBM-PIM（Aquabolt-XL）**——在 HBM2 每个 bank 边界嵌 FP16 SIMD 阵列，官方称对特定 AI 负载性能约翻倍、能耗约降一半；以及面向 CXL 的 **CXL-PNM/CMM** 系列。([Near Memory Solutions for the AI Era, Samsung](https://semiconductor.samsung.com/news-events/tech-blog/near-memory-solutions-for-the-ai-era/)、[Samsung PIM at Hot Chips 2023, ServeTheHome](https://www.servethehome.com/samsung-processing-in-memory-technology-at-hot-chips-2023/))
- **硬件压缩内存**：把压缩/解压做进内存控制器或 CXL Type-3 设备里，对软件**透明地放大有效容量**——相当于把 zram 的逻辑做成硬件。这一方向产品化程度参差，**多为厂商方案 / 研究原型，缺乏统一标准，待核实**。

这两块本质上都是"**让内存不只是存储，而能就地处理**"，与分层内存正交又互补：分层管"页放哪层"，近内存计算管"在那层能不能直接算/压"。学界综述见 [*The Landscape of Compute-near-memory and Compute-in-memory*（arXiv 2401.14428）](https://arxiv.org/pdf/2401.14428)。

## 4. 历史：为什么演变成今天这样

一条清晰的主线是**"非均匀"从硬件渗到软件**：

1. **NUMA 起步**：多插槽服务器早就有"本地/远端内存"之分，内核为此做了 NUMA 节点与 NUMA balancing——但那时只有"远近"两档、且同为 DRAM。
2. **PMEM 催生分层**：Intel Optane 持久内存把"比 DRAM 慢、比 SSD 快"的**第三档**摆到了 DRAM 之外，内核被迫从"NUMA 远近"升级为"**有高低之分的内存层**"——5.15 的 demote-on-reclaim、6.1 的显式 memory tier 都是这波的产物（[mm/demotion, LWN](https://lwn.net/Articles/901205/)）。
3. **Optane 退场、CXL 接棒**：持久内存产品线收缩后，**CXL 成了"DRAM 之外的慢内存层"的主要载体**，把同一套分层框架接了过去——内核里 memory tier 与 CXL 驱动几乎是并肩长出来的。
4. **冷热识别独立化**：分层让"挑冷页/热页"的需求暴涨，恰逢 [A05](../foundations/A05-冷热识别的演进.md) 的 **DAMON** 把访问监控做成可编程框架——两条线在 2024 年前后**正式合流**（§6.1）。

**一句话**：分层内存不是凭空设计的，而是硬件（PMEM→CXL）不断往 DRAM 与存储之间"塞新层"，逼着内核把回收/迁移从"两档 NUMA"演进成"可扩展的多层迁移"。

## 5. 现状与平台差异

| 维度 | 数据中心（Linux x86/ARM 服务器） | 移动端（Android / iOS / HarmonyOS） |
|---|---|---|
| 硬件层 | 本地 DRAM + CXL Type-3 + （历史）PMEM；池化/远端在试点 | 仅 LPDDR 单层 DRAM；**无 CXL** |
| "分层"的体现 | 真·多 NUMA 节点 + memory tier + demote/promote | **压缩内存即"准分层"**：未压 RAM → lz4 → zstd 重压 → 可选落盘（[A06](../foundations/A06-压缩与换页.md)） |
| 冷热迁移 | DAMON_RECLAIM / DAMOS_MIGRATE_{HOT,COLD}、NUMA balancing | DAMON / MGLRU 主要服务于"回收挑页"，迁移目标是 zram 而非 CXL 节点 |
| 驱动力 | 容量墙 + 资源搁浅（TCO） | 续航 + 闪存寿命 + 有限物理内存 |

> **术语警示**：服务器语境的"tier/node/demotion"是**真物理层间迁移**；移动端语境的"分层"目前主要指 [A06](../foundations/A06-压缩与换页.md) 的**压缩多级**，二者机制不同、别混用。iOS 的压缩内存（XNU compressor）更不是 Linux 的 memory tier。

## 6. 趋势与未解问题 ← 本篇重心

### 6.1 DAMON 接管分层迁移（与 A05 的合流）

§3.1 指出 NUMA-balancing 式提升**反应慢**。新方向是让 [A05 的 DAMON](../foundations/A05-冷热识别的演进.md) 直接驱动层间迁移：在 DAMOS 里新增两个动作——

- **`DAMOS_MIGRATE_COLD`**：在 DRAM 节点上**主动把冷页降级到 CXL 节点**，腾出快层；
- **`DAMOS_MIGRATE_HOT`**：在 CXL 节点上**主动把热页提升回 DRAM**；
- 配 `target_nid` sysfs 旋钮指定目标节点，并借助新的 `YOUNG` DAMOS filter 做访问判定。

SK 海力士等给出的实测：高内存压力下，相比默认 NUMA 策略，把 Redis 落在 CXL 上的量从 ~28.7GB 压到 0.4–5.5GB，**性能下降从 ~11% 收窄到 3–5%**（基于 v6.10-rc3 的 RFC，**尚未稳定合入主线，版本与措辞待核实**）。([DAMON based tiered memory management for CXL, LWN](https://lwn.net/Articles/978313/))

> 这正应了 [A05 §6.2 的判断](../foundations/A05-冷热识别的演进.md)：**MGLRU 让回收挑页更准，DAMON 让"冷热"成为可被任意策略（包括层间迁移）消费的一等公民。** 分层内存是 DAMON 当下最重要的应用场景。

### 6.2 压缩换页向"分层"靠拢

[A06 §6](../foundations/A06-压缩与换页.md) 的几条线本质都是分层化：zram `MULTI_COMP` 重压缩（lz4 热页 / zstd 冷页）、memcg 感知的 zswap shrinker、与 MGLRU 代际信息打通——都是在"**未压 → 弱压 → 强压 → 落盘/远端**"之间做更细的冷热分层。换言之，**移动端不需要 CXL，也在用压缩做"软分层"**。

### 6.3 终端视角：数据中心思路如何反哺移动端

终端短期内见不到 CXL 硬件，但**三件套——分层 + 冷热迁移 + 压缩——的方法论正在落地移动端**：

- **分层**：zram 多级（重压缩）+ 可选 writeback，就是 DRAM 内的"软多层"；
- **冷热迁移**：DAMON 的物理地址监控 + DAMOS `pageout`/`cold` 已可在 Android 内核做访问感知的主动回收（启用与机型覆盖**待核实**，见 [A05](../foundations/A05-冷热识别的演进.md)）；
- **压缩**：[A06](../foundations/A06-压缩与换页.md) 的 zram 本就是"用 CPU 换容量"的本地版近内存思路。

**一句综述判断**：CXL/远端内存这类**硬件**大概率长期停在数据中心；但其催生的**内核基础设施**（memory tier、DAMON 迁移、per-cgroup 分层公平性）是**全平台共享的代码**，会顺着 GKI 流回 Android，让移动端用"压缩 + DAMON"复刻同一套分层逻辑。

### 6.4 公认的未解问题

- **迁移的时机与抖动**：提升太慢则热页挨慢层之苦，太激进则来回搬页（thrashing 的层间版）——没有普适策略；
- **一致性与容错**：远端/共享内存里，节点故障、缓存一致开销仍是硬骨头（[disaggregation 综述](https://arxiv.org/pdf/2305.03943)）；
- **接口稳定性**：内核 memory-tiering 的 sysfs/用户接口几经反复（v9 一度删掉 sysfs 补丁，[LWN](https://lwn.net/Articles/901205/)），尚未完全定型；
- **CXL 生态的商业不确定性**：设备能否量产、软件能否跟上，业界仍有保留（[LWN](https://lwn.net/Articles/1017599/)）。

## 8. 实测 / 观测点

- `/sys/kernel/mm/numa/demotion_enabled`：开/关回收时降级；
- `/sys/devices/system/node/`：看 NUMA 节点拓扑；CXL 内存呈现为 **CPU-less 节点**（`has_cpu` 为否、有 memory）；
- `/sys/devices/virtual/memory_tiering/`（视内核版本）：内存层与各层成员节点（**接口随版本变动，待核实**）；
- `numactl -H`、`lstopo`：查看节点距离与内存分布；
- DAMON：`/sys/kernel/mm/damon/` 配置物理地址监控与 DAMOS 迁移方案（`migrate_hot`/`migrate_cold` + `target_nid`），或用户态 [`damo`](https://github.com/awslabs/damo) 工具；
- CXL：`cxl list`（cxl-cli）、`/sys/bus/cxl/` 看设备与 region；`dmesg | grep cxl`。
