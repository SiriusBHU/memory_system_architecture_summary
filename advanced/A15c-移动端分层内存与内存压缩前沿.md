## 1. 定位：移动端的"分层"长什么样

[A15](A15-前沿-先进内存.md) 把"内存"摊成一条有代价差的谱系：DRAM → 压缩内存 → CXL 扩展 → 远端内存 → 慢存储。但那条谱系的后半段——CXL、池化、远端内存——**终端一格都摸不到**：手机就一块焊死的 LPDDR，没有插槽可扩、没有 fabric 可池化、没有磁盘 swap 可落（多数 Android 出厂不挂 swapfile，iOS 根本没有传统 swap）。

于是移动端的"分层"被压进了一个很窄的区间。把 A15 的谱系在手机上重画一遍，长这样：

```
   寄存器 / L1-L3 缓存（SRAM）
   ├─ 未压缩 LPDDR（工作集，活跃页）          ← 唯一的"快层"
   ├─ zram · lz4 压缩页（温页，求快）          ┐
   ├─ zram · zstd 重压缩页（冷页，求省）        ├─ A06：全在同一块 RAM 内，靠算法分级
   └─ zram writeback → 后备块设备（idle/不可压） ┘  ← 唯一可选的"落盘"出口
```

**A15 谱系的整个后半段，在手机上坍缩成了 zram 设备内部的几级。** 本篇就讲这个坍缩：压缩内存如何成为"最靠近 CPU 的一层"、谁给它供冷热信号、为什么再往下的层进不来、以及 iOS 用什么对照方案。

## 2. 压缩内存：设备内的微型分层

[A06](../foundations/A06-压缩与换页.md) 把 zram 讲成"RAM 里的压缩块设备"。但今天的 zram **早已不是单层压缩**——它内部就是一套微型分层，三级各有代价差：

| 级 | 机制 | 角色 | 代价 |
|---|---|---|---|
| L0 | 未压缩 LPDDR | 活跃工作集 | 占满 RAM |
| L1 | **lz4 主压缩** | 刚换出的温页 | 解压快（微秒级），压缩比中等 |
| L2 | **zstd 重压缩**（`recompress`） | 久未访问的冷压缩页 | 解压慢一点，压缩比高、更省 RAM |
| L3 | **writeback 后备设备** | idle 或不可压缩页 | 一次闪存 I/O，但腾出整页 RAM |

这套"级"对应两个内核配置项，二者都已**进入 Android GKI**：

- **`CONFIG_ZRAM_MULTI_COMP`（多级重压缩）**：zram 支持"1 个主算法 + 至多 3 个次算法"，对**冷的压缩页用更强算法重压缩**（`recompress`，作用对象是 `type=idle`/`type=huge` 的页）。典型组合就是"温页用 lz4 求快、冷页用 zstd 求省"。AOSP 的 GKI 内核发行说明明确把它写进 **kernel 6.6**："新的 `CONFIG_ZRAM_MULTI_COMP` 构建项让 ZRAM 用三种备选算法之一重压缩页面……进一步缩小压缩内存、为活跃任务腾出更多空闲空间"（[Android Kernel release notes](https://source.android.com/docs/core/architecture/kernel/release-notes)、[zram 文档, kernel.org](https://docs.kernel.org/admin-guide/blockdev/zram.html)）。
- **`CONFIG_ZRAM_WRITEBACK`（写回）**：把 **idle 页或不可压缩页**写到一个后备块设备，让"压了也没用 / 久未访问"的页离开宝贵 RAM。内核文档的写回模式正好是 `idle` / `huge`（不可压缩）/ `huge_idle` / `incompressible`；为护闪存，还可用 `writeback_limit_enable` + `writeback_limit` 给写回**设预算上限**（[zram 文档, kernel.org](https://docs.kernel.org/admin-guide/blockdev/zram.html)）。

**关键认知**：移动端不需要 CXL，也已经在用压缩做"软分层"。zram 内部"未压 → lz4 → zstd → 后备设备"这条链，机制上就是 A15 里"按代价分层"的同构物——只不过层与层的差异不是物理介质（DRAM vs CXL），而是**算法强度（解压延迟 vs 压缩比）**。这是终端落地分层思想最务实的一条路。

## 3. 谁给分层供冷热信号：MGLRU + 主动回收

分层的前提是**知道哪页冷、哪页热**——往下沉的得是冷页，留在快层的得是热页。这套冷热信号在移动端主要来自两块，都已进入 Android：

### 3.1 MGLRU：把"代际"做成更准的冷热刻度

A05 讲过 MGLRU（多代 LRU）的来历：它正是 Google 的 Yu Zhao 针对 **Android 小内存设备在双链表模型下来回抖动**而设计的，把 active/inactive 两档换成多个"代（generation）"的时间带，回收永远先取最老的一代。AOSP 的 GKI 发行说明把它落在 **kernel 6.1**："MGLRU 已加入 kernel 6.1，通过更好地识别哪些页真正在用来改进内存管理"（[Android Kernel release notes](https://source.android.com/docs/core/architecture/kernel/release-notes)）。对分层而言，MGLRU 的价值是**让"该往下沉哪页"的判断更准**——代际比"两档冷热"提供了更细的刻度。

### 3.2 主动回收：在没有压力时就估工作集、预先腾挪

被动回收要等内存吃紧才动手；分层更想要**主动回收（proactive reclaim）**——在压力到来前就把冷页压进 zram、把工作集估准。两条机制要分清，**别混为一谈**：

- **cgroup v2 的 `memory.reclaim`**：一个**只写**接口，向目标 cgroup 写入字节数即触发回收（`echo 1G > memory.reclaim`），还可带 `swappiness` 参数指定本次回收的换出倾向。内核文档特别强调：**它触发的回收"不代表该 cgroup 有内存压力"**——所以不会顺带触发 socket 内存等压力联动。这是 per-memcg 主动回收的标准入口（2022 年引入，[cgroup-v2 文档, kernel.org](https://docs.kernel.org/admin-guide/cgroup-v2.html)、[per-memcg proactive reclaim, LWN](https://lwn.net/Articles/892328/)）。
- **MGLRU 自带的工作集估计 / 主动回收**：通过 `/sys/kernel/debug/lru_gen` 暴露。**务必看清**：内核文档把"工作集估计（measures how much memory an application needs in a given time interval）"和"主动回收（induces page reclaim when there is no memory pressure）"都归在 **Experimental features** 之下，**需要用户主动写命令驱动、并非开箱即用**；真正作为稳定运行时旋钮、开箱可用的只有 `/sys/kernel/mm/lru_gen/min_ttl_ms` 这个**抖动防护（thrashing prevention）**（[MGLRU 文档, kernel.org](https://docs.kernel.org/admin-guide/mm/multigen_lru.html)）。

> **一句话纠偏**：MGLRU 的宣传材料常说"工作集估计、主动回收开箱即用"——按上游文档，**那是实验性 debug 接口，需要 userspace 主动驱动**。在 Android 上，把这些信号接到一个具体的 userspace 策略（由谁、按什么节奏调 `memory.reclaim` 或 lru_gen）才是产品化的关键，而**各厂商 GKI 的实际接法与机型覆盖属于厂商定制，待核实**。

### 3.3 信号 → 动作的闭环

把两节合起来：**MGLRU 提供更细的冷热刻度，主动回收提供"压力到来前就动手"的触发**，二者共同决定"哪些温页该压进 lz4、哪些冷页该 `recompress` 成 zstd、哪些 idle 页该 writeback"。这正是 §2 那套微型分层的"调度器"。注意它和 [A08](../foundations/A08-压力与低内存终止.md) 的 **PSI + lmkd** 是两条并行的线：lmkd 是"压力撞顶就杀进程"的硬手段，主动回收是"压力到来前先腾挪"的软手段——前者保命、后者续航，分层吃的是后者。

## 4. Android 会不会再加几层？swap-to-disk 的取舍

A15 数据中心那套"再插一层慢内存"的思路，到手机上第一反应是：**那加个磁盘 swap 当最底层不行吗？** 答案是高度克制的"基本不"，原因是闪存：

- **寿命约束**：移动端 swap 落到 **UFS/eMMC**，闪存擦写次数有限。把热数据反复换进换出会显著加剧**写放大与磨损**——这正是 [A06 §2](../foundations/A06-压缩与换页.md) 列出的核心痛点。所以 Android 的默认是 **zram 完全在 RAM 内、默认不落盘**（[zram 文档, kernel.org](https://docs.kernel.org/admin-guide/blockdev/zram.html)）。一份独立的工程梳理也印证了这一点：Android 是"无盘（diskless）"路线最典型的代表——**数十亿设备跑 zram、完全不挂磁盘 swap，配一个 userspace 杀进程守护（lmkd）**（[Debunking zswap and zram myths, Chris Down](https://chrisdown.name/2026/03/24/zswap-vs-zram-when-to-use-what.html)）。
- **唯一开的口子是"有预算的 writeback"**：zram 的 `writeback` 不是"全面落盘"，而是**只把 idle / 不可压缩页**这种"留在 RAM 里也没价值"的页，在 `writeback_limit` 预算内挪到后备设备。换句话说，Android 给闪存层留的不是"第二级 swap"，而是"**给真正没价值的页一个可选的、限量的出口**"。

> **取舍的本质**：数据中心加层是为"容量墙 + 资源搁浅（TCO）"，代价是延迟非均匀；移动端拒绝加磁盘层是为"续航 + 闪存寿命"，宁可用 CPU 周期（压缩）换容量，也不愿用闪存写次数换容量。**同一套"按代价分层"的框架，目标函数完全不同**——这是终端与数据中心最根本的分野。

## 5. 为什么 CXL / 远端内存进不了手机

A15 §3 的 CXL、池化、远端内存，**短期内与终端无缘**，原因不在软件而在物理与形态：

| 维度 | 数据中心可行 | 手机为何不行 |
|---|---|---|
| **成本** | 一池内存摊给多主机，TCO 划算 | 单机单用户，多一块内存模组就是纯增本，无池化收益 |
| **形态 / 接口** | 有 PCIe 5.0/6.0 通道、可插 CXL Type-3 设备 | LPDDR 焊死、无 PCIe 内存通道、无机箱空间容纳额外模组 |
| **功耗** | 供电与散热充裕，链路常开可接受 | CXL 控制器 + 链路常开的静态功耗，对续航是净负担 |
| **跨主机** | 池化/共享的前提是多机 fabric | 终端就一台机，"跨主机借内存"无场景 |

> **但思想会反哺**：A15 §6.3 的判断在移动端依然成立——CXL/远端内存这类**硬件**大概率长期停在数据中心，但它催生的**内核基础设施**是全平台共享的代码：`memory_tier` 的显式分层、DAMON 的层间迁移动作、per-cgroup 的分层公平性，都顺着 **GKI** 流回 Android。终端不会拿到 CXL 节点，但会用"压缩多级 + DAMON 物理地址监控"复刻同一套"按代价分层、按冷热迁移"的逻辑——**硬件进不来，方法论进得来**。这也是把 zram `recompress` 看成"软 demotion"的底气。

## 6. iOS 的对照：compressed memory

把视线移到 iOS，会看到一套**思路同构、术语与代码完全不同**的方案——务必区分平台术语，**不要把 Android 的 zram sysfs 旋钮套到 iOS 上**：

| 维度 | **Android（Linux）** | **iOS / Darwin（XNU）** |
|---|---|---|
| 压缩载体 | zram 块设备（`/dev/zram0`），被当 swap 用 | **XNU VM compressor**（`vm_compressor`），不是块设备 |
| 压缩算法 | 常用 lz4（冷页可 zstd 重压缩） | **WKdm + LZ4**（XNU 内核压缩器） |
| 是否落盘 | 默认不落盘；writeback 仅给 idle/不可压页 | iOS 历史上**只压缩不落盘**（无传统磁盘 swap） |
| 杀进程侧 | **lmkd**（userspace，PSI 驱动） | **jetsam / memorystatus**（内核线程） |
| 观测口径 | `zramctl` / `/sys/block/zram0/mm_stat` | Instruments 的 "Compressed Memory" 列——**显示的 *swap* 实为压缩内存** |

核实要点：XNU 同时用 **WKdm 与 LZ4** 做页压缩；iOS 移植时面对"无 swap 可溢出"的移动约束，把"swap"做成了纯压缩内存；jetsam 则负责低内存事件下尽量回收、必要时杀进程，维持一池可回收内存以吸收瞬时尖峰（[newosxbook · Handling low memory in iOS](http://newosxbook.com/articles/MemoryPressure.html)、[xnu/doc memorystatus_notify](https://github.com/apple-oss-distributions/xnu/blob/main/doc/vm/memorystatus_notify.md)）。

**两边的同构在于结构而非实现**：都把"压缩内存"当作最靠近 CPU 的换出层、都尽量不碰闪存、都配一个低内存杀进程守护（lmkd / jetsam）。但 iOS 的 compressed memory **不是 Linux 的 memory tier、也不是 zram**——它没有 zram 那种"显式多级 recompress + writeback"的设备内分层接口（XNU 是否做等价的冷压缩页二次处理，**待核实**）。

## 7. 趋势与未解问题

移动端把分层做进一块 LPDDR，剩下的难题都绕着"**能效约束**"打转：

- **能效约束下的自适应压缩**：压缩比与解压延迟的权衡随负载漂移，没有放之四海皆准的算法选择（[A06 §6](../foundations/A06-压缩与换页.md)）。移动端还多一层约束——**压缩/解压本身耗 CPU、耗电**。"温页 lz4、冷页 zstd"的静态分级是粗糙的近似；理想是**按页冷热与当前能耗预算自适应选算法/选层**，但"何时值得花 CPU 去重压一页"在续航约束下仍无成熟策略。
- **主动回收的节奏**：主动回收太激进会无谓地压/解页、烧电并增加 refault；太保守又错过腾挪窗口。谁（userspace 哪个组件）按什么节奏调 `memory.reclaim` / lru_gen，目前是厂商各自调参的"黑魔法"，**缺乏公开的、可移植的策略基线，待核实**。
- **设备内存（dma-buf）纳入分层的难点**：手机里相当一部分内存是 [A09 设备内存](../foundations/A09-设备内存全景.md) 的 **dma-buf / gralloc 缓冲、CMA、secure heap**——GPU 纹理、相机/视频缓冲、安全堆。这些内存常被 **pinned、不可换出、不进普通 LRU**，压缩/分层这套机制**够不着它们**。如何让"设备侧的冷缓冲"也参与冷热识别与某种降级，是移动端分层最硬的一块骨头：它牵涉 dma-buf 的所有权与生命周期、IOMMU 映射、以及"压缩一块正被 GPU 共享的缓冲"在一致性上的麻烦——**目前基本是空白，待核实**。
- **DAMON 物理地址监控落地终端**：A15b 提到的 DAMON 物理地址监控 + DAMOS `pageout`/`cold`，理论上能在 Android 内核做访问感知的主动回收，把"冷页下沉"从 LRU 近似升级为真正的访问统计驱动——但**在 Android/GKI 的实际启用与机型覆盖待核实**（见 [A05](../foundations/A05-冷热识别的演进.md)、[A15b](A15b-DAMON与分层内存实践.md)）。

> **一句综述判断**：移动端的"分层前沿"不在"加几层硬件"，而在"**把已有的一块 RAM 切得更聪明**"——用更准的冷热信号（MGLRU/DAMON）驱动更细的压缩分级（recompress）与更克制的落盘（限额 writeback），且全程被续航与闪存寿命两条红线钳着。数据中心拼的是容量与利用率，终端拼的是**每一焦耳电、每一次闪存擦写换来的有效内存**。

## 9. 实测 / 观测点

- `cat /sys/block/zram0/mm_stat`：原始 vs 压缩字节、压缩比、同页计数；`bd_stat` 看 writeback 后备设备统计；
- `cat /sys/block/zram0/comp_algorithm` 与 `recomp_algorithm`：主算法 / 次算法（`MULTI_COMP` 的分级）；
- `echo idle > /sys/block/zram0/recompress`：手动触发对 idle 页的重压缩；`echo idle > /sys/block/zram0/writeback` 触发写回（受 `writeback_limit` 约束）；
- `echo 1G > /<cgroup>/memory.reclaim`（可带 `swappiness=`）：对某 cgroup 触发主动回收（cgroup v2）；
- `/sys/kernel/mm/lru_gen/min_ttl_ms`：MGLRU 抖动防护（稳定旋钮）；`/sys/kernel/debug/lru_gen`：工作集估计 / 主动回收（实验性，需主动写命令）；
- Android：`dumpsys meminfo` 看 SwapPss、`cat /proc/swaps` 确认 zram 作为唯一 swap 挂载（度量见 [A13](../foundations/A13-内存度量与排障.md)）；
- iOS / macOS：Instruments 的 VM Tracker / "Compressed Memory" 列——注意它显示的 *swap* 实为压缩内存。
