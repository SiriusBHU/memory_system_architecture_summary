> **⚠️ 全 family 立场（先读）**：A16 由 Agent 负载三特征推出的「下一代系统 × 芯片协同设计」9 个方案，**均为基于学界研究与第一性原理的设计探讨 / 可行性分析，业界（Android / iOS / HarmonyOS）尚未落地这些方案**——本 family **不是**对现有平台实现的描述。每篇都严格区分**① 已有的通用机制与学术工作**（事实、带来源，是推演的地基）与**② 本系列据此推演的设想**（非现状，标「设想 / 推测 / 待核实」）。所有「现状与平台差异」一节列的都是 ① 的**已有基础设施**。贯穿全程的核心问题是：顺着第一性原理与学界，这样做**是否成立、可行性多大**。

## 1. 定位：为什么在 A15 之外再开一个 family

A01–A13 默认一个隐含前提：**主存是一整块 DRAM，唯一的消费者是 CPU 上的进程**。A15 family 打破了"一块 DRAM"——把内存摊成 DRAM→压缩→CXL→远端的谱系。**A16 打破的是另一半前提：消费者、数据量、冷热规律。**

驱动力是**端侧 AI / Agent 负载**：本地跑起来的大模型推理、多模态编码、长上下文的 agentic 循环、RAG 向量检索、多个模型常驻待命。它带来的不是"多一个吃内存的 app"，而是**内存负载的性质变了**。A16 就是沿这条"负载先变"的线，把它对系统与芯片的连锁要求一篇篇拆开。

> A15 问"内存能摊成几层"；A16 问"**当负载变成这样，系统和芯片各自得怎么改**"。

## 2. 立论：终端的"叠加而非替换" + 三大特征

**最重要的定调（区别于数据中心）**：终端**不能为 Agent 整机专用**。数据中心可以拿整机伺候一个 LLM 推理服务；终端不行——**Agent 负载是叠加在传统终端负载之上、而非替换它**：NPU/GPU 上的权重·KV·激活，必须与传统前后台应用、系统服务、相机/视频管线**在同一块 LPDDR 上共存竞争**。这条立场贯穿全 family，每篇都回扣它。

在这条立场下，三大特征各自带上**终端独有的尖锐性**：

### 【异构】更异构的内存数据消费 —— 多任务 · 多 IP 竞争
不只是"一个 Agent 内部的数据异构（权重/KV/激活并存）"，而是**Agent 与传统负载同时竞争**有限内存与带宽：CPU 上的前后台 app、page cache、相机/视频 [dma-buf](../foundations/A09-设备内存全景.md)，与 NPU/GPU/DSP/ISP 的 AI 数据**并发消费、互不相让**。
- **系统侧裂缝**：[A09](../foundations/A09-设备内存全景.md) 的设备内存大量 **pinned、不进 LRU**；[A10](../foundations/A10-IOMMU-SMMU与DMA.md) 的 IOMMU/SVA 让设备走统一地址，但**缺页/LRU 这套冷热治理够不着 DMA 数据**。
- **芯片侧裂缝**：单一 DRAM 无法同时满足"NPU 要带宽 / CPU 要延迟 / 冷权重要容量"→ 需统一内存、PIM、异构介质协同。

### 【增长】更多层级的内存数据分布 —— KV/DMA 与传统负载双重增长
增长不止来自模型权重——**端侧 KV cache 常驻于 NPU 可访问的 dma-buf / 设备内存**（[A09](../foundations/A09-设备内存全景.md)），其增长直接挤压可回收池且**不走普通 LRU**；与此同时传统负载（app 数量、page cache、多媒体缓冲）也在涨，**两股增长叠加**。而终端**没有磁盘 swap 退路**（[A15c §4](A15c-移动端分层内存与内存压缩前沿.md)）。
- **系统侧裂缝**：[A06](../foundations/A06-压缩与换页.md) 压缩换页是当前主力，但**算法/落盘量与延迟预算的权衡无建模**；KV 这类新数据缺专门压缩。
- **芯片侧裂缝**：需新介质（MRAM 非易失多级缓存）、新存储形态（HBF 把 HBM 式堆叠带宽下放端侧）、压缩 IP。

### 【冷热】更复杂的内存冷热特征 —— 前后台皆热 · 长程后台任务
经典启发式"**前台热、后台冷**"在 Agent 时代失效——**后台长程 agent 任务（持续推理/检索/工具调用）让后台也热**，前台交互与后台任务**同时活跃**；且 KV 的注意力稀疏、权重的按层/按专家激活让冷热**更快、更结构化、与"哪个 app 在前台"无关**。难再用"场景"粗分冷热。
- **系统侧裂缝**：[A05](../foundations/A05-冷热识别的演进.md) 的 LRU/MGLRU 是**被动近似** → 需主动扫描、可编程回收（eBPF）。
- **芯片侧裂缝**：冷热全靠软件采样太慢 → 需硬件辅助访问统计 / page fault 协同。

> 一句话立论：**负载叠加（终端约束）× 三特征（异构/增长/冷热），共同把"软件栈"和"硬件/介质"同时顶到墙上——下一代只能是系统与芯片协同地改。**

## 4. 下一代系统设计 · 四题导览（A16a–A16d）

- **[A16a · LRU 主动扫描](A16a-LRU主动扫描.md)（冷热）**：当"前台热/后台冷"失效，回收要从**被动等水线**转向**主动按节奏扫**。讲 MGLRU aging 的页表走查、DAMON 区域采样两种便宜扫描器，以及 Senpai/DAMOS 式以 PSI 为反馈的激进度自调。
- **[A16b · eBPF 可编程回收策略](A16b-eBPF可编程回收策略-Android.md)（冷热）**：再进一步，把"判冷热、保护谁"的**策略本身**做成可编程。锚点放 Android：eBPF 在 Android 已是生产级（用于网络/tracing/GPU，**非内存回收**），而用 eBPF 编程回收（eBPF-mm / Cache-is-King / BPF-OOM / reclaim_ext）仍是上游研究、**无一进主线**；并以 `sched_ext`（6.12 已落地）为"核心策略可经 BPF struct_ops 可编程化"的先例。鸿蒙另走 XVM 路线，资料稀薄、本系列不展开。
- **[A16c · 异构压缩 CDSD](A16c-异构压缩CDSD.md)（增长）**：增长把压缩顶成主力扩容，但**一把算法压一切**在异构数据上太亏。讲按数据类（匿名/文件/KV/设备缓冲）异构选压；CDSD 是终端的 LZ4 家族变体实例（规格待核实）；并厘清 KV 的**有损**专用压缩与 zram **无损**压缩是两套世界。
- **[A16d · 压缩 IP 边际建模](A16d-压缩IP边际建模.md)（增长）**：落盘量、算法固定后，**压多少、压多狠**是个控制题。把压缩路径建成有代价的服务队列，给出"边际收益 = 边际成本"的判据（原创框架、公式示意），并指出 `swappiness` 即今天那个粗糙的静态影子价格。

## 5. 下一代芯片设计 · 五题导览（A16e–A16i）

- **[A16e · IOMMU 统一内存 + 异构 PF/LRU](A16e-IOMMU统一内存与异构PF-LRU.md)（异构）**：多 IP 同块 LPDDR 竞争、且 DMA 数据当前 pinned 不受冷热治理。承 [A10 SVA](../foundations/A10-IOMMU-SMMU与DMA.md) 与 [A09 设备内存](../foundations/A09-设备内存全景.md)，讲统一内存 + 设备缺页/LRU 想把"设备侧数据"也纳入治理。
- **[A16f · 端侧 KV Cache 管理](A16f-端侧KV-Cache管理方案.md)（异构+增长 · 桥接）**：全 family 最能体现终端立场的一篇。端侧 KV 常驻 dma-buf/设备内存，既增长又 pinned、不走普通 LRU，还与传统 app 竞争——这正是端侧 KV 管理与数据中心 PagedAttention 的最大分野。
- **[A16g · DRAM、PIM 异构协同](A16g-DRAM-PIM异构协同管理.md)（异构）**：承 [A15 §3.4](A15-前沿-先进内存.md) 的 HBM-PIM 起头，讲 DRAM 与近内存/存内计算介质如何被 OS 统一调度（多为研究方向）。
- **[A16h · STT/SOT-MRAM 多级缓存](A16h-STT-SOT-MRAM多级缓存方案.md)（增长）**：非易失磁存储作缓存层，往 [00 存储金字塔](../foundations/00-内存系统总览.md) 里插新层（端侧产品化属研究阶段）。
- **[A16i · 端侧 UFS–HBF 增强](A16i-端侧UFS-HBF增强.md)（增长）**：HBF（High-Bandwidth Flash，2025 厂商提出的 HBM 式堆叠 NAND）下放到端侧 UFS，给增长压力一个更快的闪存后端（最新、最前瞻）。

- **部分需推断/旁证 🟡**：[A16c](A16c-异构压缩CDSD.md)（KV 压缩论文充足，CDSD 缩写本身待核实）、[A16g](A16g-DRAM-PIM异构协同管理.md)（硬件有、OS 统一调度多研究）、[A16h](A16h-STT-SOT-MRAM多级缓存方案.md)（器件有、端侧产品化待证）。
